# Primeiros DAGs en Airflow

Despois de revisar os conceptos básicos, o papel de Airflow no proxecto e a interface web, o seguinte paso natural é empezar a escribir DAGs pequenos e executables. Neste capítulo o obxectivo non é construír fluxos complexos, senón entender a sintaxe básica, a estrutura dun ficheiro DAG e algúns patróns moi habituais.

## Como se crea e se rexistra un DAG en Airflow

Antes de entrar na estrutura interna dun DAG, convén entender o proceso práctico máis básico: como pasa de ser un ficheiro Python a converterse nun DAG visible dentro de Airflow.

En termos simples, o procedemento adoita ser este:

- crear un ficheiro `.py` no directorio de DAGs configurado no contorno
- escribir nel a definición do DAG
- gardar o ficheiro e agardar a que Airflow o procese
- comprobar na UI ou na CLI se o DAG foi rexistrado correctamente

Airflow non require un paso manual de importación desde a interface. O habitual é que os servizos do sistema revisen periodicamente o directorio de DAGs, analicen os ficheiros Python e, se non hai erros, incorporen o novo DAG á páxina de `DAGs`.

### Fluxo básico

Unha secuencia típica de traballo sería esta:

1. crear un novo ficheiro Python no directorio designado para os DAGs
2. definir nel un DAG sinxelo
3. gardar os cambios
4. esperar uns segundos a que o contorno procese o ficheiro
5. abrir a páxina de `DAGs` e comprobar se aparece

Nos primeiros contactos con Airflow, este ciclo é importante porque axuda a relacionar directamente tres elementos:

- o ficheiro `.py`
- o procesado interno que fai Airflow
- a visualización final na interface

### Como comprobar se o DAG aparece

A forma máis directa de ver se todo foi ben é abrir a interface web e buscar o novo DAG na páxina de `DAGs`.

Se o ficheiro se procesou correctamente, o normal é que:

- o DAG apareza na listaxe
- poida activarse ou pausarse
- poida lanzarse con `Trigger DAG`

Tamén pode facerse unha comprobación desde a liña de comandos. Por exemplo, unha forma habitual de ver os DAGs rexistrados é empregar:

```bash
airflow dags list
```

Neste proxecto, ese comando executaríase habitualmente dentro do contedor `airflow-scheduler`, que é un dos compoñentes principais do stack de Airflow.

Unha forma directa de acceder ao contedor sería:

```bash
docker exec -it airflow-scheduler bash
```

Unha vez dentro, xa pode lanzarse:

```bash
airflow dags list
```

Esta comprobación resulta útil porque permite verificar se o DAG foi cargado mesmo antes de inspeccionalo con detalle na UI.

### Que facer se non aparece ou se ten erros

Se o DAG non aparece, o primeiro non é asumir que Airflow fallou, senón revisar algúns puntos básicos:

- confirmar que o ficheiro está no directorio correcto
- comprobar se hai erros de sintaxe no `.py`
- agardar un pequeno intervalo por se o procesado aínda non rematou
- revisar o servizo `airflow-dag-processor`

Na interface tamén poden verse indicios útiles. Se o DAG aparece pero non funciona correctamente, convén:

- abrir o DAG concreto
- revisar a execución correspondente
- entrar nos `Logs` da task que fallou

Desde a CLI tamén poden empregarse comandos de apoio para diagnosticar problemas de carga ou validación. Segundo o caso, poden resultar útiles ordes como:

```bash
airflow dags list
airflow dags show dag_id
airflow tasks test dag_id task_id 2026-01-01
```

De novo, estas comprobacións faríanse normalmente dentro de `airflow-scheduler` despois de entrar no contedor coa orde anterior.

Máis adiante veremos exemplos completos, pero desde o principio interesa quedar cunha idea práctica: un DAG existe para Airflow só cando o ficheiro Python está ben formado, pode ser procesado sen erros e o sistema é capaz de rexistralo.

## Que elementos básicos ten un DAG

Antes de ver exemplos completos, convén situar que ingredientes adoitan aparecer nun DAG sinxelo. Nun primeiro contacto, o normal é atopar estes elementos:

- imports necesarios
- definición do `DAG`
- parámetros básicos como `dag_id`, `start_date`, `schedule`, `catchup` e `tags`
- definición das tarefas
- dependencias entre tarefas, se hai máis dunha

Visto de forma moi simple, un ficheiro DAG adoita comezar cos imports, continúa coa definición do `DAG`, e dentro dese bloque decláranse as tarefas e as súas dependencias.

Isto significa que un DAG non é só unha tarefa illada, senón unha pequena definición composta por configuración, tarefas e relacións entre elas.

## Sintaxe básica dun ficheiro DAG

Un DAG sinxelo adoita definirse nun ficheiro Python que importa Airflow, crea o obxecto `DAG` e declara as tarefas correspondentes.

Un exemplo mínimo de sintaxe podería ser este:

```python
from datetime import datetime

from airflow import DAG

with DAG(
    dag_id="dag_minimo_demo",
    start_date=datetime(2026, 1, 1),
    schedule=None,
    catchup=False,
    tags=["demo"],
) as dag:
    pass
```

Este exemplo aínda non fai nada útil, pero serve para introducir varios elementos fundamentais da sintaxe:

- `dag_id`: identifica o DAG dentro de Airflow
- `start_date`: marca o punto de partida temporal do DAG
- `schedule=None`: indica que non se executa automaticamente
- `catchup=False`: evita execucións pendentes do pasado
- `tags`: permite organizar mellor os DAGs na interface

A nivel didáctico, este é un bo primeiro paso porque permite ver a forma básica dun DAG antes de meter operadores e tarefas reais.

## Un primeiro DAG mínimo visible na interface

Unha vez entendida a estrutura mínima, xa pode empregarse ese mesmo patrón para crear un DAG moi pequeno que Airflow sexa capaz de rexistrar e mostrar na interface.

```python
from datetime import datetime

from airflow import DAG

with DAG(
    dag_id="dag_minimo_demo",
    start_date=datetime(2026, 1, 1),
    schedule=None,
    catchup=False,
    tags=["demo"],
) as dag:
    pass
```

Este exemplo é útil por dous motivos:

- permite comprobar que o ficheiro se procesa correctamente
- permite verificar que o DAG aparece na páxina de `DAGs`

## Tarefas con `PythonOperator`

Un dos primeiros elementos reais que adoita introducirse nun DAG é unha tarefa Python. Para iso emprégase habitualmente `PythonOperator`.

### Sintaxe básica

A forma básica é esta:

```python
from airflow.operators.python import PythonOperator


def nome_da_funcion():
    ...


nome_da_tarefa = PythonOperator(
    task_id="nome_da_tarefa",
    python_callable=nome_da_funcion,
)
```

Aquí aparecen dúas pezas fundamentais:

- `task_id`: identifica a tarefa dentro do DAG
- `python_callable`: indica que función Python debe executarse

### Exemplo

```python
from datetime import datetime

from airflow import DAG
from airflow.operators.python import PythonOperator


def saudar():
    print("Ola desde Airflow")


with DAG(
    dag_id="hello_airflow",
    start_date=datetime(2026, 1, 1),
    schedule=None,
    catchup=False,
    tags=["demo", "python"],
) as dag:
    tarefa_saudo = PythonOperator(
        task_id="tarefa_saudo",
        python_callable=saudar,
    )
```

Este exemplo xa permite comprobar algo moi importante:

- unha tarefa pode executar unha función Python
- a saída de `print(...)` poderá verse nos `Logs`

## Dependencias entre tarefas

Cando un DAG ten máis dunha tarefa, é necesario expresar a orde entre elas. Airflow permite facelo de varias maneiras, e convén coñecelas desde o principio.

### Sintaxe básica

As formas máis habituais son estas:

```python
tarefa_1 >> tarefa_2
```

```python
tarefa_2 << tarefa_1
```

```python
tarefa_1 >> tarefa_2 >> tarefa_3
```

```python
[tarefa_1, tarefa_2] >> tarefa_3
```

Todas expresan dependencias, pero con lecturas lixeiramente distintas:

- `tarefa_1 >> tarefa_2`: `tarefa_1` vai antes
- `tarefa_2 << tarefa_1`: significa o mesmo, pero lido ao revés
- `tarefa_1 >> tarefa_2 >> tarefa_3`: crea unha cadea de dependencias
- `[tarefa_1, tarefa_2] >> tarefa_3`: varias tarefas deben rematar antes da seguinte

### Exemplo

```python
from datetime import datetime

from airflow import DAG
from airflow.operators.python import PythonOperator


def preparar():
    print("Preparando datos")


def procesar():
    print("Procesando datos")


with DAG(
    dag_id="dependencias_demo",
    start_date=datetime(2026, 1, 1),
    schedule=None,
    catchup=False,
    tags=["demo", "dependencias"],
) as dag:
    tarefa_preparar = PythonOperator(
        task_id="tarefa_preparar",
        python_callable=preparar,
    )

    tarefa_procesar = PythonOperator(
        task_id="tarefa_procesar",
        python_callable=procesar,
    )

    tarefa_preparar >> tarefa_procesar
```

Neste caso:

- primeiro execútase `tarefa_preparar`
- despois execútase `tarefa_procesar`

## Tarefas con `BashOperator`

Outro elemento moi habitual é `BashOperator`, que permite executar comandos do sistema. Isto reforza a idea de que Airflow non executa só Python.

### Sintaxe básica

```python
from airflow.operators.bash import BashOperator


nome_da_tarefa = BashOperator(
    task_id="nome_da_tarefa",
    bash_command="comando_a_executar",
)
```

O parámetro esencial é:

- `bash_command`: o comando de shell que se quere lanzar

### Exemplo

```python
from datetime import datetime

from airflow import DAG
from airflow.operators.bash import BashOperator

with DAG(
    dag_id="bash_demo_airflow",
    start_date=datetime(2026, 1, 1),
    schedule=None,
    catchup=False,
    tags=["demo", "bash"],
) as dag:
    tarefa_echo = BashOperator(
        task_id="tarefa_echo",
        bash_command="echo 'Executando unha tarefa bash en Airflow'",
    )

    tarefa_data = BashOperator(
        task_id="tarefa_data",
        bash_command="date",
    )

    tarefa_echo >> tarefa_data
```

Este exemplo axuda a ver que:

- Airflow orquestra tarefas
- esas tarefas poden chamar tamén a comandos do sistema

## Parámetros adicionais do DAG

Ademais da estrutura mínima, moitos DAGs incorporan algúns parámetros adicionais que melloran a súa claridade na interface.

### Sintaxe básica

```python
with DAG(
    dag_id="exemplo",
    description="Descrición do DAG",
    start_date=datetime(2026, 1, 1),
    schedule=None,
    catchup=False,
    tags=["demo"],
) as dag:
    ...
```

Aquí resultan especialmente útiles:

- `description`: engade unha explicación breve do DAG
- `tags`: facilita a organización e filtrado na UI

### Exemplo

```python
from datetime import datetime

from airflow import DAG
from airflow.operators.python import PythonOperator


def inicio():
    print("Inicio do fluxo")


def transformacion():
    print("Transformación intermedia")


def fin():
    print("Fin do fluxo")


with DAG(
    dag_id="dag_tres_tarefas_demo",
    description="Exemplo sinxelo cun fluxo de tres tarefas",
    start_date=datetime(2026, 1, 1),
    schedule=None,
    catchup=False,
    tags=["demo", "tres-tarefas"],
) as dag:
    tarefa_inicio = PythonOperator(
        task_id="tarefa_inicio",
        python_callable=inicio,
    )

    tarefa_transformacion = PythonOperator(
        task_id="tarefa_transformacion",
        python_callable=transformacion,
    )

    tarefa_fin = PythonOperator(
        task_id="tarefa_fin",
        python_callable=fin,
    )

    tarefa_inicio >> tarefa_transformacion >> tarefa_fin
```

Neste exemplo xa aparecen:

- `description`
- varias tarefas
- dependencias encadeadas
- nomes claros para o DAG e para cada task

## Como se verán estes DAGs na interface

Unha vez gardado un ficheiro DAG no cartafol correspondente, Airflow debería detectalo, procesalo e mostralo na interface web.

No caso do primeiro exemplo, o comportamento esperado sería este:

- o DAG aparece na páxina de `DAGs`
- pode lanzarse con `Trigger DAG`
- a execución xera un `dag run`
- cada tarefa mostra o seu estado
- os `Logs` permiten ver a saída concreta da execución

É dicir, o código e a interface están directamente conectados:

- o ficheiro Python define o fluxo
- a UI permite observalo e operalo

## Onde mirar se algo non aparece ou non funciona

Nos primeiros exemplos adoitan aparecer algúns problemas frecuentes. Por iso convén ter claro desde o principio onde mirar.

Se un DAG non aparece:

- revisar se o ficheiro Python está no cartafol correcto
- comprobar se hai erros de sintaxe
- mirar o servizo `airflow-dag-processor`

Se o DAG aparece pero falla:

- abrir a task correspondente
- revisar os `Logs`
- comprobar que a función ou o comando executado son correctos

Un erro simple de sintaxe pode impedir que Airflow rexistre o DAG. Por exemplo:

```python
with DAG(
    dag_id="dag_con_erro",
    start_date=datetime(2026, 1, 1),
    schedule=None,
    catchup=False
) as dag
    pass
```

Neste caso falta `:` ao final da liña `as dag`, e iso impediría que o ficheiro se procesase correctamente.

## Boas prácticas mínimas para empezar

Ao escribir os primeiros DAGs, convén manter algunhas regras simples:

- usar `dag_id` e `task_id` claros e descritivos
- empezar con DAGs pequenos
- empregar `schedule=None` nas primeiras probas
- usar `catchup=False` en contornos docentes e locais
- comprobar sempre os `Logs` antes de sacar conclusións

Tamén é recomendable non meter demasiada lóxica complexa no propio ficheiro do DAG. Ao principio interesa máis entender ben a estrutura do fluxo que construír exemplos demasiado ambiciosos.

## Para seguir avanzando

Unha vez dominados estes exemplos iniciais, o seguinte paso natural é introducir DAGs algo máis expresivos:

- paso de datos simples entre tarefas
- uso de XCom

Nese punto, Airflow deixa de verse só como unha colección de exemplos pequenos e empeza a funcionar como unha ferramenta real de orquestración dentro do stack do proxecto.
