# NiFi controlado desde Airflow

Apache NiFi encaixa no pipeline como ferramenta de inxesta, movemento e primeira preparacion de datos.

Este documento explica como incorporar fluxos de NiFi xa deseñados e validados dentro dun pipeline orquestrado por Airflow. A parte específica de NiFi, como o uso da interface, os procesadores, os `Process Groups`, os `Parameter Contexts`, os `Controller Services` e as integracións con sistemas externos, desenvolvese nos apuntamentos de NiFi do curso:

- https://github.com/adbgonzalez/nifi

Un caso de uso natural no stack do laboratorio e:

```text
MongoDB Atlas -> NiFi -> MinIO bronze -> Spark silver/gold -> PostgreSQL/Superset
```

Neste escenario:

- NiFi le datos desde MongoDB Atlas.
- NiFi escribe os datos crus ou semipreparados en MinIO, dentro da capa `bronze`.
- Airflow coordina cando debe executarse ese fluxo.
- Spark continua o pipeline lendo desde `bronze` e xerando `silver` e `gold`.

## Papel de cada ferramenta

E importante manter separadas as responsabilidades.

NiFi non substitue Airflow, e Airflow non substitue NiFi.

NiFi e axeitado para:

- conectarse a fontes externas
- extraer datos desde APIs, bases de datos ou ficheiros
- mover datos entre sistemas
- facer transformacións lixeiras
- aplicar enrutado, validación e control de fluxo
- escribir no lago de datos

Airflow e axeitado para:

- decidir cando se executa o pipeline
- establecer dependencias entre pasos
- lanzar ou comprobar fluxos externos
- rexistrar historico de execucions
- coordinar NiFi, Spark, Kafka, PostgreSQL ou Superset dentro dun mesmo DAG

O punto de partida e un fluxo de NiFi xa existente e validado. Airflow non crea procesadores nin configura o canvas de NiFi en cada execucion; a sua función e controlar, supervisar e encadear ese fluxo co resto do DAG.

## Fluxo NiFi de partida

O exemplo empregado nesta guía parte dun `Process Group` exportado desde NiFi co nome:

```text
mongorecordread
```

O fluxo le documentos desde MongoDB Atlas, aplica unha conversión baseada na Record API, agrupa a saída e escribe o resultado en MinIO.

O deseño do `Process Group` é:

```text
GetMongo -> ConvertRecord -> MergeContent -> PutS3Object
```

Este fluxo emprega compoñentes tratados nos apuntamentos de NiFi:

- `GetMongo` le documentos desde MongoDB Atlas.
- `ConvertRecord` normaliza ou converte o formato dos rexistros.
- `MergeContent` xunta FlowFiles nunha saída común.
- `PutS3Object` escribe o resultado en MinIO.

O `Process Group` emprega un `Parameter Context` chamado:

```text
atlas
```

Cos seguintes parámetros funcionais:

```text
mongo.db = probas
mongo.col = lecturas_test100
```

A conexión con Atlas aparece parametrizada mediante:

```text
mongo.uri
```

Ese parámetro debe tratarse como sensible. Se un export de NiFi inclúe unha URI de MongoDB con usuario e contrasinal, convén rexenerar ou cambiar esa credencial e evitar publicar o ficheiro exportado sen limpalo antes.

O fluxo tamén referencia dous `Controller Services` externos:

```text
MongoDBControllerService
AWSCredentialsProviderControllerService
```

E dous `Controller Services` internos para Record API:

```text
JsonTreeReader
JsonRecordSetWriter
```

Antes de controlar o fluxo desde Airflow, estes servizos deben estar correctamente configurados e habilitados en NiFi.

### Lectura desde MongoDB Atlas

O procesador `GetMongo` está configurado para ler da base e colección indicadas polo `Parameter Context`:

```text
Mongo Database Name = #{mongo.db}
Mongo Collection Name = #{mongo.col}
```

Neste caso:

```text
Base de datos: probas
Colección: lecturas_test100
```

A consulta filtra documentos coa orixe:

```json
{
  "orixe": "nifi-record"
}
```

E a proxección limita os campos exportados:

```json
{
  "_id": 0,
  "sensor": 1,
  "valor": 1,
  "ts": 1,
  "orixe": 1
}
```

Ademais, o fluxo limítase a lectura a:

```text
Limit = 100
Batch Size = 100
Results Per FlowFile = 100
```

Esta configuración é cómoda para unha proba controlada. Nunha carga real podería eliminarse ou parametrizarse ese límite.

Outro detalle importante é a planificación de `GetMongo`:

```text
Scheduling Strategy = TIMER_DRIVEN
Run Schedule = 1 sec
```

Isto significa que, mentres o procesador estea en execucion, NiFi pode lanzar a consulta repetidamente. Para unha invocacion controlada desde Airflow, cómpre evitar que o grupo quede correndo máis tempo do necesario ou que a mesma consulta xere datos duplicados.

### Escritura en MinIO

O procesador `PutS3Object` escribe contra MinIO mediante:

```text
Endpoint Override URL = http://minio:9000
Bucket = mongo-export
Content Type = application/json
```

A clave do obxecto está definida como:

```text
mongo/#{mongo.col}/${now():format("yyyy/MM/dd/HHmmss")}.json
```

Co valor actual de `mongo.col`, a saída terá unha forma semellante a:

```text
mongo-export/mongo/lecturas_test100/2026/05/05/123000.json
```

Desde o punto de vista da arquitectura Medallion, esa ruta pode interpretarse como zona de entrada ou `bronze`. Se se quere facelo máis explícito, podería adaptarse a unha estrutura deste estilo:

```text
mongo-export/bronze/lecturas_test100/yyyy/MM/dd/HHmmss.json
```

ou ben:

```text
datalake/bronze/lecturas_test100/dt=yyyy-MM-dd/<ficheiro>.json
```

O importante para Airflow é coñecer a ruta onde NiFi deixa os datos, porque esa será a entrada que se comprobe antes de lanzar Spark.

### Agrupacion dos rexistros

Entre `ConvertRecord` e `PutS3Object` aparece un procesador `MergeContent`.

Neste fluxo está configurado cunha estratexia de tipo:

```text
Merge Strategy = Bin-Packing Algorithm
Merge Format = Binary Concatenation
Minimum Number of Entries = 1
Maximum Number of Entries = 1000
Demarcator = salto de liña
```

Isto permite xuntar a saída xerada polo paso de conversión antes de escribila en MinIO. O resultado é útil como ficheiro JSON ou como ficheiro con rexistros separados por liñas, dependendo de como chegue a saída de `ConvertRecord`.

## Como encaixa no DAG

O DAG de Airflow non deberia conter a loxica interna de NiFi. Esa loxica xa vive no canvas de NiFi ou nun fluxo versionado.

O DAG pode limitarse a este patrón:

```text
pedir_token
  -> preparar_process_group_nifi
  -> executar_processor_fonte_unha_vez
  -> esperar_a_que_nifi_remate
  -> parar_process_group_nifi
  -> comprobar_bronze
  -> spark_bronze_to_silver
  -> spark_silver_to_gold
```

O punto clave e que Airflow usa NiFi como un sistema externo con API REST.

Neste exemplo non se arrinca todo o `Process Group` de golpe para deixalo correndo. O patrón probado foi:

- arrancar os procesadores descendentes de `GetMongo`
- executar `GetMongo` unha soa vez con `RUN_ONCE`
- esperar a que haxa actividade real e a que despois queden as colas baleiras
- deter o `Process Group` ao final, tamén se algunha tarefa falla

## Conexion HTTP de Airflow con NiFi

No stack do laboratorio, NiFi esta dispoñible dentro da rede Docker co nome:

```text
nifi
```

E o servizo HTTPS interno escoita no porto:

```text
8443
```

En Airflow pode crearse unha conexion HTTP co identificador:

```text
nifi_default
```

Configuracion recomendada:

```text
Conn Type: HTTP
Schema: https
Host: nifi
Port: 8443
```

No campo `Extra`:

```json
{
  "verify": false
}
```

Para probas locais, `verify: false` evita problemas co certificado HTTPS autoxerado de NiFi. Nun contorno real deberia empregarse un certificado válido.

Se se quere gardar tamén usuario e contrasinal na conexion, poden empregarse os campos:

```text
Login
Password
```

Deben ser as credenciais de NiFi, non as de MongoDB Atlas. Deste xeito evítase escribir credenciais directamente no DAG.

Tamén se pode crear coa CLI de Airflow:

```bash
airflow connections add nifi_default \
  --conn-type http \
  --conn-schema https \
  --conn-host nifi \
  --conn-port 8443 \
  --conn-login <usuario_nifi> \
  --conn-password '<contrasinal_nifi>' \
  --conn-extra '{"verify": false}'
```

## Variables de Airflow necesarias

Ademais da conexión `nifi_default`, o DAG necesita dúas `Variables` de Airflow.

| Variable | Valor no cluster probado | Significado |
| --- | --- | --- |
| `nifi_mongorecordread_pg_id` | `1d83aa03-019c-1000-7759-2d1c952b97e3` | ID real da instancia do `Process Group` `mongorecordread` |
| `nifi_mongorecordread_source_processor_id` | `1d9dc67a-019c-1000-f6df-b6a7a245f622` | ID real da instancia do procesador fonte `GetMongo` |

Estas variables créanse desde a interface de Airflow en:

```text
Administración -> Variables -> +
```

Tamén poden crearse coa CLI de Airflow:

```bash
airflow variables set nifi_mongorecordread_pg_id 1d83aa03-019c-1000-7759-2d1c952b97e3
airflow variables set nifi_mongorecordread_source_processor_id 1d9dc67a-019c-1000-f6df-b6a7a245f622
```

Non é necesario crear variables de Airflow para `mongo.db`, `mongo.col` ou `mongo.uri`. Eses valores pertencen ao `Parameter Context` `atlas` dentro de NiFi. Airflow só necesita saber que grupo debe controlar e que procesador é a fonte que se executará unha vez.

Se o fluxo se importa noutra instancia de NiFi, estes valores poden cambiar. Nese caso hai que volver copiar os IDs reais desde a interface de NiFi e actualizar as variables de Airflow.

## Autenticacion contra NiFi

NiFi expón unha API REST baixo:

```text
/nifi-api
```

Para obter un token con usuario e contrasinal, o endpoint é:

```text
POST /nifi-api/access/token
```

O contido envíase como formulario:

```text
username=<usuario>&password=<contrasinal>
```

A resposta e un token de texto que despois se usa no header:

```text
Authorization: Bearer <token>
```

Este patrón encaixa co uso de `HttpOperator` e `PythonOperator` descrito no capítulo de operadores específicos.

## Identificar o Process Group

Para controlar un fluxo desde Airflow fai falta coñecer o identificador do `Process Group` de NiFi.

Hai varias formas de obtelo:

- abrir o `Process Group` na interface de NiFi e revisar a URL
- seleccionar o grupo e mirar os detalles do componente
- consultar a API de NiFi e buscar o grupo polo seu nome

Na práctica docente, o máis sinxelo é copiar o ID desde a interface de NiFi e gardalo nunha Variable de Airflow:

```text
nifi_mongorecordread_pg_id
```

No export do `Process Group` aparecen dous identificadores:

```text
identifier = 8a84a446-ad89-3b32-9892-3553954cb5f7
instanceIdentifier = 1d83aa03-019c-1000-7759-2d1c952b97e3
```

Para chamadas á API REST de NiFi debe empregarse o identificador da instancia real do compoñente no NiFi onde se executa o fluxo. No export corresponde ao campo `instanceIdentifier`, non ao campo `identifier` do fluxo versionado.

Se o fluxo se importa noutro NiFi, ese ID pode cambiar. Por iso é mellor gardalo como Variable de Airflow e non escribilo directamente no código do DAG.

O nome funcional do grupo é:

```text
mongorecordread
```

Pero as chamadas á API adoitan necesitar o ID, non só o nome.

O DAG probado tamén necesita identificar o procesador fonte, porque ese procesador se executa unha soa vez con `RUN_ONCE`. No export, `GetMongo` aparece así:

```text
processor name = GetMongo
identifier = 93d69ff9-97a9-303d-b320-33cd24b2a684
instanceIdentifier = 1d9dc67a-019c-1000-f6df-b6a7a245f622
```

Ese `instanceIdentifier` é o valor que debe gardarse en:

```text
nifi_mongorecordread_source_processor_id
```

## Arrancar e parar un Process Group

NiFi permite programar ou deter os compoñentes dun `Process Group` mediante a API:

```text
PUT /nifi-api/flow/process-groups/{id}
```

Para arrancar os compoñentes autorizados do grupo:

```json
{
  "id": "<process_group_id>",
  "state": "RUNNING"
}
```

Para detelos:

```json
{
  "id": "<process_group_id>",
  "state": "STOPPED"
}
```

Isto non crea nin modifica o deseño do fluxo. Só cambia o estado de execucion dos compoñentes.

Para este fluxo concreto, a execucion máis controlada non consiste en poñer todo o grupo en `RUNNING` desde o principio. Como `GetMongo` ten unha planificación interna, o DAG arranca primeiro os procesadores que están despois da fonte e logo lanza só o procesador `GetMongo` en modo:

```text
RUN_ONCE
```

Para cambiar o estado dun procesador individual emprégase:

```text
PUT /nifi-api/processors/{processor_id}/run-status
```

Antes desa chamada cómpre ler o procesador con:

```text
GET /nifi-api/processors/{processor_id}
```

A resposta inclúe a `revision` actual, que NiFi require no corpo do `PUT`. O DAG fai esa lectura antes de cada cambio de estado.

Se os `Controller Services` do grupo non estan habilitados, pode ser necesario habilitalos antes de arrancar os procesadores. Nese caso, a API correspondente é:

```text
PUT /nifi-api/flow/process-groups/{id}/controller-services
```

co estado:

```json
{
  "id": "<process_group_id>",
  "state": "ENABLED"
}
```

Nun laboratorio, o máis cómodo e deixar os `Controller Services` xa habilitados desde NiFi e que Airflow só cambie o estado dos procesadores ou deteña o `Process Group`.

## Consultar o estado dun Process Group

Para saber se o fluxo segue traballando, Airflow pode consultar:

```text
GET /nifi-api/flow/process-groups/{id}/status?recursive=true
```

Esta chamada devolve estatísticas do grupo e dos seus compoñentes, como:

- FlowFiles en cola
- bytes en cola
- fíos activos
- FlowFiles de entrada e saída
- estado dos procesadores descendentes

Para unha execucion controlada desde Airflow, unha comprobacion práctica pode combinar:

- non hai fíos activos
- non quedan FlowFiles en cola
- houbo algunha actividade desde que empezou a execucion

Isto non substitue unha validacion dos datos en MinIO, pero serve para detectar se NiFi segue procesando.

## Exemplo de DAG probado

O seguinte exemplo mostra o DAG usado neste laboratorio, que segue o seguinte fluxo de traballo:

1. pide un token a NiFi
2. le os contadores actuais do `Process Group`
3. arranca os procesadores descendentes de `GetMongo`
4. executa `GetMongo` unha soa vez con `RUN_ONCE`
5. espera a que NiFi teña actividade e despois quede sen colas nin fíos activos
6. para o `Process Group` aínda que a espera falle

```python
from __future__ import annotations

import logging
import time
from datetime import datetime

import requests
from airflow.providers.standard.operators.python import PythonOperator
from airflow.sdk import DAG, Variable
from airflow.sdk.bases.hook import BaseHook


logger = logging.getLogger(__name__)

STATUS_COUNTER_FIELDS = (
    "flowFilesIn",
    "flowFilesOut",
    "flowFilesTransferred",
    "bytesIn",
    "bytesOut",
    "bytesTransferred",
    "bytesRead",
    "bytesWritten",
    "taskCount",
)


def int_status(value) -> int:
    if value in (None, ""):
        return 0
    if isinstance(value, (int, float)):
        return int(value)
    return int(str(value).replace(",", ""))


def nifi_connection():
    conn = BaseHook.get_connection("nifi_default")
    verify = conn.extra_dejson.get("verify", True)
    base_url = f"{conn.schema}://{conn.host}:{conn.port}/nifi-api"
    return conn, base_url, verify


def pedir_token():
    conn, base_url, verify = nifi_connection()

    response = requests.post(
        f"{base_url}/access/token",
        headers={"Content-Type": "application/x-www-form-urlencoded"},
        data={
            "username": conn.login,
            "password": conn.password,
        },
        verify=verify,
        timeout=30,
    )
    response.raise_for_status()
    return response.text


def process_group_id():
    return Variable.get("nifi_mongorecordread_pg_id")


def source_processor_id():
    return Variable.get("nifi_mongorecordread_source_processor_id")


def processor_name(processor: dict) -> str:
    return processor.get("component", {}).get("name") or processor.get("id", "unknown")


def status_process_group(base_url: str, verify, token: str, pg_id: str):
    response = requests.get(
        f"{base_url}/flow/process-groups/{pg_id}/status",
        headers={"Authorization": f"Bearer {token}"},
        params={"recursive": "true"},
        verify=verify,
        timeout=30,
    )
    response.raise_for_status()

    status = response.json()["processGroupStatus"]
    snapshot = status["aggregateSnapshot"]
    return status, snapshot


def counters_from_snapshot(snapshot: dict) -> dict[str, int]:
    return {field: int_status(snapshot.get(field)) for field in STATUS_COUNTER_FIELDS}


def counters_advanced(current: dict[str, int], baseline: dict[str, int]) -> bool:
    return any(current[field] > int_status(baseline.get(field)) for field in STATUS_COUNTER_FIELDS)


def cambiar_estado_process_group(state: str, ti=None):
    _, base_url, verify = nifi_connection()
    token = ti.xcom_pull(task_ids="pedir_token")
    pg_id = process_group_id()

    response = requests.put(
        f"{base_url}/flow/process-groups/{pg_id}",
        headers={
            "Authorization": f"Bearer {token}",
            "Content-Type": "application/json",
        },
        json={
            "id": pg_id,
            "state": state,
        },
        verify=verify,
        timeout=30,
    )
    response.raise_for_status()


def listar_processors_process_group(base_url: str, verify, token: str, pg_id: str) -> list[dict]:
    response = requests.get(
        f"{base_url}/process-groups/{pg_id}/processors",
        headers={"Authorization": f"Bearer {token}"},
        verify=verify,
        timeout=30,
    )
    response.raise_for_status()
    return response.json().get("processors", [])


def cambiar_estado_processor(base_url: str, verify, token: str, processor_id: str, state: str):
    processor_response = requests.get(
        f"{base_url}/processors/{processor_id}",
        headers={"Authorization": f"Bearer {token}"},
        verify=verify,
        timeout=30,
    )
    processor_response.raise_for_status()
    revision = processor_response.json()["revision"]

    response = requests.put(
        f"{base_url}/processors/{processor_id}/run-status",
        headers={
            "Authorization": f"Bearer {token}",
            "Content-Type": "application/json",
        },
        json={
            "revision": revision,
            "state": state,
            "disconnectedNodeAcknowledged": False,
        },
        verify=verify,
        timeout=30,
    )
    response.raise_for_status()


def preparar_process_group(ti=None):
    _, base_url, verify = nifi_connection()
    token = ti.xcom_pull(task_ids="pedir_token")
    pg_id = process_group_id()
    source_id = source_processor_id()
    _, snapshot = status_process_group(base_url, verify, token, pg_id)
    baseline = counters_from_snapshot(snapshot)

    for processor in listar_processors_process_group(base_url, verify, token, pg_id):
        processor_id = processor["id"]
        component = processor.get("component", {})
        state = component.get("state")

        if processor_id == source_id or state == "DISABLED":
            continue

        logger.info("Arrincando processor NiFi: %s (%s)", processor_name(processor), processor_id)
        cambiar_estado_processor(base_url, verify, token, processor_id, "RUNNING")

    return baseline


def executar_processor_fonte_unha_vez(ti=None):
    _, base_url, verify = nifi_connection()
    token = ti.xcom_pull(task_ids="pedir_token")
    processor_id = source_processor_id()

    logger.info("Executando unha vez o processor fonte NiFi: %s", processor_id)
    cambiar_estado_processor(base_url, verify, token, processor_id, "RUN_ONCE")


def parar_process_group(ti=None):
    cambiar_estado_process_group("STOPPED", ti=ti)


def esperar_a_que_nifi_remate(ti=None):
    _, base_url, verify = nifi_connection()
    token = ti.xcom_pull(task_ids="pedir_token")
    pg_id = process_group_id()
    baseline = ti.xcom_pull(task_ids="preparar_process_group_nifi") or {}

    deadline = time.monotonic() + 10 * 60
    work_seen = False

    while time.monotonic() < deadline:
        status, snapshot = status_process_group(base_url, verify, token, pg_id)
        counters = counters_from_snapshot(snapshot)

        queued = int_status(snapshot.get("flowFilesQueued"))
        active_threads = int_status(snapshot.get("activeThreadCount"))
        run_status = snapshot.get("runStatus") or status.get("runStatus") or "UNKNOWN"
        work_seen = (
            work_seen
            or queued > 0
            or active_threads > 0
            or counters_advanced(counters, baseline)
        )

        logger.info(
            "Estado NiFi: run_status=%s queued=%s active_threads=%s "
            "transferred=%s in=%s out=%s tasks=%s work_seen=%s",
            run_status,
            queued,
            active_threads,
            counters["flowFilesTransferred"],
            counters["flowFilesIn"],
            counters["flowFilesOut"],
            counters["taskCount"],
            work_seen,
        )

        if queued == 0 and active_threads == 0 and work_seen:
            cambiar_estado_process_group("STOPPED", ti=ti)
            return

        if queued == 0 and active_threads == 0 and run_status.lower() == "stopped":
            return

        time.sleep(10)

    raise TimeoutError(
        "NiFi non rematou dentro do tempo esperado. "
        "Revisa nos logs as liñas 'Estado NiFi' para ver queued, active_threads e transferred."
    )


with DAG(
    dag_id="nifi_mongorecordread_minio_demo",
    start_date=datetime(2026, 1, 1),
    schedule=None,
    catchup=False,
    tags=["nifi", "atlas", "minio", "mongo"],
) as dag:
    tarefa_token = PythonOperator(
        task_id="pedir_token",
        python_callable=pedir_token,
    )

    tarefa_preparar_nifi = PythonOperator(
        task_id="preparar_process_group_nifi",
        python_callable=preparar_process_group,
    )

    tarefa_executar_fonte = PythonOperator(
        task_id="executar_processor_fonte_unha_vez",
        python_callable=executar_processor_fonte_unha_vez,
    )

    tarefa_esperar_nifi = PythonOperator(
        task_id="esperar_a_que_nifi_remate",
        python_callable=esperar_a_que_nifi_remate,
    )

    tarefa_parar_nifi = PythonOperator(
        task_id="parar_process_group_nifi",
        python_callable=parar_process_group,
        trigger_rule="all_done",
    )

    tarefa_token >> tarefa_preparar_nifi >> tarefa_executar_fonte >> tarefa_esperar_nifi >> tarefa_parar_nifi
```

Neste exemplo, `parar_process_group_nifi` usa `trigger_rule="all_done"` para intentar deter o fluxo aínda que a espera falle. Ademais, `esperar_a_que_nifi_remate` tamén chama a `STOPPED` cando detecta que xa houbo traballo e que non quedan colas nin fíos activos.

Nun DAG real, despois de `parar_process_group_nifi` poderían vir tarefas como:

```text
comprobar_bronze
spark_bronze_to_silver
spark_silver_to_gold
gold_to_postgres
```

## Comprobar que hai datos en bronze

Non chega con saber que NiFi deixou de ter colas. Tamén interesa comprobar que os datos chegaron a MinIO.

Unha estratexia sinxela é comprobar o prefixo de saída esperado.

No fluxo `mongorecordread`, a saída configurada en `PutS3Object` é:

```text
s3a://mongo-export/mongo/lecturas_test100/
```

ou, desde o punto de vista de MinIO:

```text
bucket: mongo-export
prefix: mongo/lecturas_test100/
```

Esta comprobacion pode facerse de varias formas:

- cun sensor de Airflow compatible con S3/MinIO
- cunha tarefa Python usando `boto3`
- cunha tarefa Spark que falle se non atopa datos de entrada
- cunha comprobacion manual durante unha primeira demo

Para un pipeline avaliable, o ideal é que haxa unha tarefa explícita de comprobacion antes de lanzar Spark.

## Continuacion con Spark

Unha vez que NiFi escribiu a capa `bronze`, Spark pode continuar co pipeline.

O patrón seria:

```text
NiFi escribe bronze
Spark le bronze
Spark escribe silver
Spark escribe gold
```

Nun DAG, isto encaixa con `SparkSubmitOperator`:

```python
nifi_bronze >> comprobar_bronze >> bronze_to_silver >> silver_to_gold
```

Onde:

- `nifi_bronze` representa a parte controlada por NiFi
- `comprobar_bronze` valida que hai datos de entrada
- `bronze_to_silver` lanza un job Spark
- `silver_to_gold` lanza outro job Spark

## Boas practicas

Para integrar NiFi con Airflow convén seguir algunhas pautas:

- Deseñar e probar primeiro o fluxo en NiFi.
- Encapsular o fluxo nun `Process Group`.
- Usar nomes claros para os grupos, por exemplo `mongorecordread` ou `atlas_lecturas_to_minio`.
- Gardar endpoints, buckets e nomes de colección en `Parameter Contexts`.
- Gardar credenciais sensibles en mecanismos seguros.
- Usar Airflow para coordinar, non para reconstruír o canvas de NiFi.
- Evitar que o mesmo fluxo xere datos duplicados se se executa varias veces.
- Escribir en rutas particionadas por data ou por execucion.
- Comprobar a chegada de datos a `bronze` antes de lanzar Spark.
- Deixar evidencias nos logs de Airflow e na Provenance de NiFi.

## Limitacions

Arrancar un `Process Group` desde Airflow non significa exactamente "executa unha vez e remata".

NiFi esta deseñado como motor de fluxo continuo. Moitos procesadores traballan por planificación interna, por eventos ou por colas.

Por iso, cando se usa desde Airflow, hai que deseñar o fluxo con coidado:

- Se o fluxo e continuo, quizais non teña sentido arrincalo e paralo en cada DAG run.
- Se o fluxo representa unha carga puntual, debe quedar claro cando se considera rematado.
- Se `GetMongo` ten unha consulta fixa e unha planificación de `0 sec`, pode repetir a mesma lectura mentres o grupo estea activo.
- Se `GetMongo` le sempre os mesmos documentos, poden aparecer duplicados en `bronze`.
- Se a saída en MinIO non inclúe partición por data ou execucion, pódense sobrescribir datos.

Nunha arquitectura real, adoita ser mellor que NiFi estea executando fluxos de inxesta de maneira estable e que Airflow coordine as fases posteriores. Pero para unha práctica docente, controlar un `Process Group` desde Airflow permite ver moi ben como se integran ambas ferramentas.

## Resumo

NiFi pode encargarse da inxesta desde MongoDB Atlas e da escritura en MinIO como capa `bronze`.

Airflow pode actuar por riba como orquestrador:

1. autentica contra NiFi
2. arranca ou comproba o `Process Group`
3. espera a que a inxesta remate
4. valida a chegada de datos a MinIO
5. lanza Spark para procesar `silver` e `gold`

Esta integración reforza a idea central do proxecto: cada ferramenta fai a parte para a que está mellor preparada, e Airflow coordina o fluxo completo.
