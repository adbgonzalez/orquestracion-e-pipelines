# Comunicación entre tarefas en Airflow

No capítulo anterior vimos como crear DAGs sinxelos, definir tarefas e establecer dependencias entre elas. O seguinte paso natural é entender como pode compartirse información entre distintas tarefas dentro dun mesmo fluxo.

É importante diferenciar dúas ideas que ás veces se confunden ao empezar:

- unha dependencia indica orde de execución
- un intercambio de datos indica que unha tarefa produce información que outra utiliza

É dicir, escribir `tarefa_1 >> tarefa_2` significa que `tarefa_2` debe esperar a que remate `tarefa_1`, pero non implica por si só que os datos dunha pasen á outra.

## Que é `XCom`

En Airflow, o mecanismo máis habitual para pasar valores pequenos entre tarefas é `XCom`, abreviatura de `cross-communication`.

De forma sinxela, `XCom` permite:

- gardar un valor durante a execución dunha tarefa
- recuperar ese valor desde outra tarefa do mesmo DAG

Este mecanismo é útil cando se queren compartir resultados pequenos, por exemplo:

- unha mensaxe
- un número
- unha ruta de ficheiro
- un identificador xerado nun paso previo

En cambio, non está pensado para mover grandes volumes de datos. Se un fluxo necesita compartir ficheiros pesados ou estruturas grandes, o habitual é gardalos nun sistema externo e pasar entre tarefas só a referencia necesaria.

## Como se produce e se recupera un valor

Nun caso básico, unha tarefa pode devolver un valor con `return`, e Airflow gárdao automaticamente como `XCom`. Despois, outra tarefa pode recuperalo durante a súa execución.

### Sintaxe básica

```python
def tarefa_produtora():
    return "valor xerado"


def tarefa_consumidora(ti=None):
    valor = ti.xcom_pull(task_ids="tarefa_produtora")
    print(valor)
```

Nesta sintaxe aparecen dúas ideas importantes:

- `return` permite que a tarefa produtora deixe un valor dispoñible
- `ti.xcom_pull(...)` permite recuperar ese valor desde outra tarefa

Aquí, `ti` fai referencia ao contexto de execución da task instance actual.

### Exemplo

```python
from datetime import datetime

from airflow import DAG
from airflow.operators.python import PythonOperator


def xerar_mensaxe():
    return "Ola desde a primeira tarefa"


def mostrar_mensaxe(ti=None):
    mensaxe = ti.xcom_pull(task_ids="xerar_mensaxe")
    print(f"Mensaxe recibida: {mensaxe}")


with DAG(
    dag_id="xcom_demo",
    start_date=datetime(2026, 1, 1),
    schedule=None,
    catchup=False,
    tags=["demo", "xcom"],
) as dag:
    tarefa_xerar = PythonOperator(
        task_id="xerar_mensaxe",
        python_callable=xerar_mensaxe,
    )

    tarefa_mostrar = PythonOperator(
        task_id="mostrar_mensaxe",
        python_callable=mostrar_mensaxe,
    )

    tarefa_xerar >> tarefa_mostrar
```

Neste exemplo:

- a primeira tarefa devolve unha cadea de texto
- Airflow gárdaa como `XCom`
- a segunda tarefa recupéraa e móstraa nos `Logs`

## Relación entre dependencias e `XCom`

Ao principio convén insistir nunha idea clave: as dependencias e o intercambio de datos están relacionados, pero non son o mesmo.

### Sintaxe básica

```python
tarefa_1 >> tarefa_2
```

Esta sintaxe só expresa orde.

Se ademais se quere recuperar un valor da tarefa anterior, hai que facer algo máis, por exemplo:

```python
valor = ti.xcom_pull(task_ids="tarefa_1")
```

### Exemplo

```python
from datetime import datetime

from airflow import DAG
from airflow.operators.python import PythonOperator


def calcular_numero():
    return 25


def mostrar_resultado(ti=None):
    numero = ti.xcom_pull(task_ids="calcular_numero")
    print(f"O número recibido é {numero}")


with DAG(
    dag_id="dependencia_e_xcom_demo",
    start_date=datetime(2026, 1, 1),
    schedule=None,
    catchup=False,
    tags=["demo", "xcom", "dependencias"],
) as dag:
    tarefa_calculo = PythonOperator(
        task_id="calcular_numero",
        python_callable=calcular_numero,
    )

    tarefa_mostrar = PythonOperator(
        task_id="mostrar_resultado",
        python_callable=mostrar_resultado,
    )

    tarefa_calculo >> tarefa_mostrar
```

Neste caso:

- a dependencia garante que a segunda tarefa non se execute antes da primeira
- `XCom` permite que a segunda recupere o valor producido pola primeira

## Que tipo de información convén pasar

Ao traballar con `XCom`, interesa manter exemplos e usos sinxelos.

Resulta razoable pasar:

- cadeas curtas
- números
- pequenos dicionarios
- nomes de ficheiros
- rutas
- identificadores

En cambio, é mellor evitar:

- ficheiros completos
- grandes coleccións de datos
- obxectos complexos innecesarios

Isto é importante porque Airflow non está pensado como sistema principal de transporte de datos, senón como ferramenta de orquestración.

## Como comprobar o valor na práctica

Unha vez executado o DAG, o máis habitual é revisar os `Logs` da tarefa consumidora para comprobar que o valor foi recibido correctamente.

Nun fluxo coma os anteriores, o comportamento esperado sería este:

- a tarefa produtora remata correctamente
- a tarefa consumidora execútase despois
- nos `Logs` da segunda aparece o valor recuperado

Se algo non funciona como se esperaba, convén revisar:

- que a dependencia entre tarefas estea ben definida
- que o `task_id` usado en `xcom_pull(...)` sexa correcto
- que a tarefa produtora devolva realmente un valor

## Boas prácticas iniciais con `XCom`

Ao comezar, convén manter algunhas regras simples:

- usar `XCom` só para valores pequenos
- empregar `task_id` claros e consistentes
- comprobar os `Logs` da tarefa que recupera o valor
- non confundir orde de execución con intercambio de datos

Tamén é recomendable empezar con exemplos moi pequenos antes de combinar `XCom` con fluxos máis complexos.

## Para seguir avanzando

Unha vez entendido este mecanismo básico, o seguinte paso natural pode ser algún destes:

- traballar con `XCom` de forma algo máis explícita
- escribir resultados en ficheiros e pasar só a ruta entre tarefas
- lanzar procesos externos e recuperar identificadores ou estados
- integrar Airflow con Spark dentro do stack do proxecto

Nese punto, os DAGs deixan de ser só secuencias de tarefas e pasan a comportarse como fluxos que coordinan operacións e intercambian información dun xeito controlado.
