# Comunicación entre tarefas en Airflow

No capítulo anterior vimos como crear DAGs sinxelos, definir tarefas e establecer dependencias entre elas. O seguinte paso natural é entender como se relacionan as tarefas dun fluxo máis alá da simple orde de execución.

Ao empezar con Airflow convén diferenciar varias ideas que ás veces se mesturan:

- unha dependencia indica que unha tarefa debe esperar por outra
- un intercambio de datos indica que unha tarefa produce información que outra utiliza
- un parámetro permite configurar unha execución
- unha conexión ou unha variable permiten reutilizar configuración sen embutir todo no código

Noutras palabras, escribir `tarefa_1 >> tarefa_2` significa que `tarefa_2` non debe executarse antes de que remate `tarefa_1`, pero iso non implica por si só que os datos dunha pasen á outra.

Esta distinción é especialmente importante neste proxecto. Moitas tarefas de Airflow actuarán como capa de orquestración sobre procesos de Spark, e os resultados reais do pipeline adoitan gardarse en sistemas externos como táboas Delta ou Iceberg. Por iso, a comunicación entre tarefas non sempre consiste en mover datos dentro de Airflow, senón en coordinar pasos, compartir pequenas referencias e controlar o fluxo.

## Dependencias fronte a intercambio de información

Airflow permite expresar moi ben a orde entre tarefas, pero a orde non é o mesmo que o intercambio de información.

Se escribimos:

```python
tarefa_1 >> tarefa_2
```

o que estamos dicindo é:

- `tarefa_1` debe executarse antes
- `tarefa_2` só poderá comezar cando a anterior remate segundo a lóxica definida

Isto describe a estrutura do fluxo, pero non indica como accede `tarefa_2` ao resultado producido por `tarefa_1`.

Nun pipeline real poden darse varias situacións distintas:

- unha tarefa produce un valor pequeno e pásao a outra
- unha tarefa escribe un ficheiro ou unha táboa externa, e a seguinte limítase a ler esa saída
- unha tarefa non precisa datos da anterior, pero si depende de que rematase correctamente
- unha tarefa decide que rama do fluxo debe continuar

Por iso, falar de comunicación entre tarefas en Airflow implica falar non só de `XCom`, senón tamén de contexto, parámetros, configuración reutilizable e control do fluxo.

## `XCom`

En Airflow, o mecanismo máis habitual para pasar valores pequenos entre tarefas é `XCom`, abreviatura de `cross-communication`.

De forma sinxela, `XCom` permite:

- gardar un valor durante a execución dunha tarefa
- recuperar ese valor desde outra tarefa do mesmo DAG

Este mecanismo é útil cando se queren compartir resultados pequenos, por exemplo:

- unha mensaxe
- un número
- unha ruta de ficheiro
- un identificador xerado nun paso previo

En cambio, non está pensado para mover grandes volumes de datos. Se un fluxo necesita compartir ficheiros pesados ou estruturas grandes, o habitual é gardalos nun sistema externo e pasar entre tarefas só a referencia necesaria.

No contexto deste proxecto, isto é especialmente relevante. Se unha tarefa de Spark escribe unha táboa Delta ou Iceberg, o normal non é gardar os datos en `XCom`, senón deixar os datos no almacenamento correspondente e, se fai falta, compartir só un nome de táboa, unha ruta ou unha partición.

## Como se produce e se recupera un valor con `XCom`

Nun caso básico, unha tarefa pode devolver un valor con `return`, e Airflow gárdao automaticamente como `XCom`. Despois, outra tarefa pode recuperalo durante a súa execución.

### Sintaxe básica

```python
def tarefa_produtora():
    return "valor xerado"


def tarefa_consumidora(ti=None):
    valor = ti.xcom_pull(task_ids="tarefa_produtora")
    print(valor)
```

Nesta sintaxe aparecen dúas ideas importantes:

- `return` permite que a tarefa produtora deixe un valor dispoñible
- `ti.xcom_pull(...)` permite recuperar ese valor desde outra tarefa

Aquí, `ti` fai referencia ao contexto de execución da task instance actual.

### Exemplo

```python
from datetime import datetime

from airflow import DAG
from airflow.operators.python import PythonOperator


def xerar_mensaxe():
    return "Ola desde a primeira tarefa"


def mostrar_mensaxe(ti=None):
    mensaxe = ti.xcom_pull(task_ids="xerar_mensaxe")
    print(f"Mensaxe recibida: {mensaxe}")


with DAG(
    dag_id="xcom_demo",
    start_date=datetime(2026, 1, 1),
    schedule=None,
    catchup=False,
    tags=["demo", "xcom"],
) as dag:
    tarefa_xerar = PythonOperator(
        task_id="xerar_mensaxe",
        python_callable=xerar_mensaxe,
    )

    tarefa_mostrar = PythonOperator(
        task_id="mostrar_mensaxe",
        python_callable=mostrar_mensaxe,
    )

    tarefa_xerar >> tarefa_mostrar
```

Neste exemplo:

- a primeira tarefa devolve unha cadea de texto
- Airflow gárdaa como `XCom`
- a segunda tarefa recupéraa e móstraa nos `Logs`

## Relación entre dependencias e `XCom`

Ao principio convén insistir nunha idea clave: as dependencias e o intercambio de datos están relacionados, pero non son o mesmo.

### Sintaxe básica

```python
tarefa_1 >> tarefa_2
```

Esta sintaxe só expresa orde.

Se ademais se quere recuperar un valor da tarefa anterior, hai que facer algo máis, por exemplo:

```python
valor = ti.xcom_pull(task_ids="tarefa_1")
```

### Exemplo

```python
from datetime import datetime

from airflow import DAG
from airflow.operators.python import PythonOperator


def calcular_numero():
    return 25


def mostrar_resultado(ti=None):
    numero = ti.xcom_pull(task_ids="calcular_numero")
    print(f"O número recibido é {numero}")


with DAG(
    dag_id="dependencia_e_xcom_demo",
    start_date=datetime(2026, 1, 1),
    schedule=None,
    catchup=False,
    tags=["demo", "xcom", "dependencias"],
) as dag:
    tarefa_calculo = PythonOperator(
        task_id="calcular_numero",
        python_callable=calcular_numero,
    )

    tarefa_mostrar = PythonOperator(
        task_id="mostrar_resultado",
        python_callable=mostrar_resultado,
    )

    tarefa_calculo >> tarefa_mostrar
```

Neste caso:

- a dependencia garante que a segunda tarefa non se execute antes da primeira
- `XCom` permite que a segunda recupere o valor producido pola primeira

## Cando usar `XCom` e cando non

Ao traballar con `XCom`, interesa manter exemplos e usos sinxelos.

Resulta razoable pasar:

- cadeas curtas
- números
- pequenos dicionarios
- nomes de ficheiros
- rutas
- identificadores

En cambio, é mellor evitar:

- ficheiros completos
- grandes coleccións de datos
- resultados de Spark
- obxectos complexos innecesarios

Isto é importante porque Airflow non está pensado como sistema principal de transporte de datos, senón como ferramenta de orquestración.

Nun proxecto como este, un patrón máis natural adoita ser:

- unha tarefa Spark escribe unha saída en Delta ou Iceberg
- a seguinte tarefa depende da anterior
- e, se fai falta, usa un valor pequeno como a ruta, o nome da táboa ou unha partición concreta

## Comunicación mediante ficheiros

Outra forma sinxela de entender a comunicación entre tarefas é empregar un ficheiro como artefacto intermedio.

Neste caso:

- unha tarefa escribe un ficheiro
- outra tarefa le ese ficheiro
- Airflow encárgase de coordinar a orde de execución

Este patrón é didácticamente útil porque deixa ver con claridade que Airflow non ten por que transportar directamente os datos entre tarefas. Ás veces o que fai é coordinar tarefas que comparten unha saída gardada nun sistema externo, aínda que ese sistema sexa algo tan simple como un ficheiro no disco.

No proxecto real, esta idea adoita materializarse en táboas Delta ou Iceberg máis que en ficheiros soltos, pero o modelo mental é moi parecido:

- unha tarefa produce unha saída persistente
- outra tarefa consome esa saída máis tarde

### Exemplo básico

```python
from datetime import datetime
from pathlib import Path

from airflow import DAG
from airflow.operators.python import PythonOperator


RUTA_FICHEIRO = Path("/tmp/airflow_demo_ficheiro.txt")


def escribir_ficheiro():
    RUTA_FICHEIRO.write_text("Ola desde a primeira tarefa", encoding="utf-8")


def ler_ficheiro():
    contido = RUTA_FICHEIRO.read_text(encoding="utf-8")
    print(f"Contido recibido: {contido}")


with DAG(
    dag_id="ficheiros_demo",
    start_date=datetime(2026, 1, 1),
    schedule=None,
    catchup=False,
    tags=["demo", "ficheiros"],
) as dag:
    tarefa_escribir = PythonOperator(
        task_id="escribir_ficheiro",
        python_callable=escribir_ficheiro,
    )

    tarefa_ler = PythonOperator(
        task_id="ler_ficheiro",
        python_callable=ler_ficheiro,
    )

    tarefa_escribir >> tarefa_ler
```

Neste exemplo:

- a primeira tarefa crea un ficheiro cun contido pequeno
- a segunda tarefa le ese mesmo ficheiro
- a comunicación real entre tarefas non se produce por `XCom`, senón mediante un recurso externo compartido

Isto axuda a preparar a idea de que, en fluxos máis reais, as tarefas poden comunicarse a través de saídas persistentes como ficheiros, táboas ou obxectos almacenados fóra de Airflow.

## Parámetros de execución

Ademais de intercambiar resultados entre tarefas, Airflow permite definir parámetros que condicionan como se executa un DAG.

Os `params` serven para introducir valores de configuración asociados ao propio fluxo ou a unha execución concreta. Isto é útil cando queremos que un mesmo DAG poida executarse con pequenas variacións sen cambiar o código principal.

A diferenza das `Variables` e das `Connections`, os `params` non se administran normalmente desde a sección `Administración` da interface. O habitual é definilos no propio código do DAG e, nalgúns casos, tamén poden fornecerse ao lanzar unha execución concreta.

Por exemplo, un parámetro pode representar:

- unha data de traballo
- un nome de contorno
- unha ruta lóxica
- un nome de táboa de destino

### Exemplo básico

```python
from datetime import datetime

from airflow import DAG
from airflow.operators.python import PythonOperator


def mostrar_parametros(**context):
    pais = context["params"]["pais"]
    print(f"País recibido: {pais}")


with DAG(
    dag_id="params_demo",
    start_date=datetime(2026, 1, 1),
    schedule=None,
    catchup=False,
    params={"pais": "galicia"},
    tags=["demo", "params"],
) as dag:
    tarefa_params = PythonOperator(
        task_id="mostrar_parametros",
        python_callable=mostrar_parametros,
    )
```

Neste caso, o parámetro non vén doutra tarefa. Forma parte da configuración coa que o DAG se executa.

Isto axuda a distinguir mellor dúas cousas:

- `XCom` serve para compartir pequenos resultados entre tarefas
- `params` serve para configurar a execución do fluxo

## Variables de Airflow

As `Variables` de Airflow permiten gardar valores reutilizables a nivel da instancia. Son útiles cando varios DAGs necesitan consultar unha configuración común sen repetir ese valor no código.

Na práctica, estas variables adoitan configurarse desde a interface web, na sección:

- `Administración` -> `Variables`

Por exemplo, unha variable pode gardar:

- un nome de bucket
- unha ruta base
- un identificador de contorno
- un valor de configuración compartido

Isto resulta útil porque separa a lóxica do DAG de certos detalles de configuración.

### Exemplo básico

```python
from datetime import datetime

from airflow import DAG
from airflow.models import Variable
from airflow.operators.python import PythonOperator


def mostrar_bucket():
    bucket = Variable.get("raw_bucket")
    print(f"Bucket configurado: {bucket}")


with DAG(
    dag_id="variables_demo",
    start_date=datetime(2026, 1, 1),
    schedule=None,
    catchup=False,
    tags=["demo", "variables"],
) as dag:
    tarefa_variable = PythonOperator(
        task_id="mostrar_bucket",
        python_callable=mostrar_bucket,
    )
```

Nun enfoque didáctico, convén introducir as variables como mecanismo de configuración compartida, non como lugar onde gardar información transitoria de execución.

É dicir:

- unha variable non substitúe `XCom`
- unha variable non representa unha conexión a un sistema
- unha variable serve para gardar valores reutilizables e relativamente estables

## Connections

Outra peza importante de Airflow son as `Connections`. Unha conexión serve para gardar a información necesaria para acceder a un sistema externo.

Na práctica, as conexións adoitan configurarse desde a interface web, na sección:

- `Administración` -> `Connections`

Pode representar, por exemplo:

- unha base de datos
- un endpoint HTTP
- un servizo cloud
- unha configuración de acceso a Spark

O importante é entender que unha `Connection` non está pensada para transportar datos entre tarefas, senón para separar do código os detalles de acceso a outros sistemas.

Isto axuda a evitar prácticas pouco recomendables, como:

- embutir credenciais directamente no DAG
- repetir endpoints en varios ficheiros
- mesturar a lóxica do fluxo coa configuración de acceso

No contexto deste proxecto, isto encaixa moi ben co uso de compoñentes externos como Spark, Hadoop, MinIO ou outros servizos do stack. O DAG debería describir o fluxo, mentres que os detalles de conexión deberían quedar o máis separados posible.

### Exemplo básico

```python
from datetime import datetime

from airflow import DAG
from airflow.hooks.base import BaseHook
from airflow.operators.python import PythonOperator


def mostrar_conexion():
    conn = BaseHook.get_connection("spark_default")
    print(f"Host configurado: {conn.host}")


with DAG(
    dag_id="connections_demo",
    start_date=datetime(2026, 1, 1),
    schedule=None,
    catchup=False,
    tags=["demo", "connections"],
) as dag:
    tarefa_connection = PythonOperator(
        task_id="mostrar_conexion",
        python_callable=mostrar_conexion,
    )
```

Neste caso, o código non define a conexión. Limítase a recuperar unha conexión xa configurada en Airflow a partir do seu identificador.

## Templates e valores dinámicos

Airflow permite que moitos campos dos operadores sexan templados. Isto significa que poden incorporar valores dinámicos usando `Jinja`.

Esta capacidade é moi útil porque permite construír comandos, rutas ou mensaxes que dependan do contexto de execución sen ter que hardcodear todo manualmente.

Por exemplo, poden templarse valores como:

- datas
- `run_id`
- parámetros
- variables de Airflow

### Exemplo básico

```python
from datetime import datetime

from airflow import DAG
from airflow.operators.bash import BashOperator


with DAG(
    dag_id="templates_demo",
    start_date=datetime(2026, 1, 1),
    schedule=None,
    catchup=False,
    params={"contorno": "dev"},
    tags=["demo", "templates"],
) as dag:
    tarefa_echo = BashOperator(
        task_id="tarefa_echo",
        bash_command="echo 'Run {{ run_id }} no contorno {{ params.contorno }}'",
    )
```

Neste exemplo, o comando non é completamente fixo. Airflow renderiza o valor final no momento da execución empregando o contexto dispoñible.

Desde un punto de vista práctico, isto é moi útil para:

- construír rutas con datas
- xerar comandos adaptados á execución actual
- reutilizar un mesmo DAG en varios contextos

## Contexto de execución

Os `templates` e varias funcións Python poden acceder ao contexto de execución do DAG. Este contexto contén información útil sobre a execución actual.

Entre os elementos que adoitan resultar máis prácticos están:

- `run_id`
- `params`
- `dag_run`
- `task`
- `ti`

Segundo o caso, este contexto pode empregarse para:

- recuperar `XCom`
- consultar parámetros
- identificar a execución actual
- construír nomes ou rutas dinámicas

Aquí convén insistir nunha idea importante: non toda a información útil dentro dunha task ten por que chegar desde outra task. Moitas veces xa existe un contexto de execución que Airflow pon á nosa disposición.

### Exemplo básico

```python
from datetime import datetime

from airflow import DAG
from airflow.operators.python import PythonOperator


def mostrar_contexto(**context):
    print(f"run_id: {context['run_id']}")
    print(f"dag_id: {context['dag'].dag_id}")
    print(f"task_id: {context['task'].task_id}")


with DAG(
    dag_id="contexto_demo",
    start_date=datetime(2026, 1, 1),
    schedule=None,
    catchup=False,
    tags=["demo", "contexto"],
) as dag:
    tarefa_contexto = PythonOperator(
        task_id="mostrar_contexto",
        python_callable=mostrar_contexto,
    )
```

Neste exemplo, a tarefa non necesita recibir datos dunha tarefa anterior. Limítase a consultar información que Airflow xa pon no contexto da execución actual.

## Branching e decisións dentro do fluxo

Ata agora vimos fluxos nos que todas as tarefas avanzan nunha orde fixa. Pero Airflow tamén permite expresar decisións: segundo unha condición, o DAG pode seguir por unha rama ou por outra.

Isto coñécese habitualmente como `branching`.

A idea xeral é esta:

- unha tarefa avalía unha condición
- esa tarefa decide que tarefa ou tarefas posteriores deben continuar
- o resto das ramas poden quedar sen executar nesa run

Desde o punto de vista conceptual, isto é moi importante porque introduce un control do fluxo máis rico que a simple cadea lineal.

Nun exemplo real, unha decisión podería depender de:

- se existe ou non unha saída previa
- se hai datos novos
- se unha validación foi correcta
- se se debe continuar cunha rama normal ou cunha rama alternativa

Aquí non fai falta aínda entrar en moitos operadores específicos. O importante neste capítulo é entender a idea: algunhas tarefas non só producen resultados, senón que tamén condicionan o camiño que seguirá o DAG.

### Exemplo básico

```python
from datetime import datetime

from airflow import DAG
from airflow.operators.empty import EmptyOperator
from airflow.operators.python import BranchPythonOperator


def escoller_rama(**context):
    executar_carga = context["params"]["executar_carga"]
    if executar_carga:
        return "tarefa_cargar"
    return "tarefa_omitir"


with DAG(
    dag_id="branching_demo",
    start_date=datetime(2026, 1, 1),
    schedule=None,
    catchup=False,
    params={"executar_carga": True},
    tags=["demo", "branching"],
) as dag:
    tarefa_decision = BranchPythonOperator(
        task_id="tarefa_decision",
        python_callable=escoller_rama,
    )

    tarefa_cargar = EmptyOperator(task_id="tarefa_cargar")
    tarefa_omitir = EmptyOperator(task_id="tarefa_omitir")
    tarefa_fin = EmptyOperator(task_id="tarefa_fin")

    tarefa_decision >> [tarefa_cargar, tarefa_omitir]
    tarefa_cargar >> tarefa_fin
    tarefa_omitir >> tarefa_fin
```

Neste exemplo:

- `tarefa_decision` avalía un parámetro
- se `executar_carga` é `True`, continúa por `tarefa_cargar`
- se non, continúa por `tarefa_omitir`

Así vese con claridade que unha tarefa pode non só executar unha acción, senón tamén decidir que rama do DAG seguirá activa.

## Trigger rules e coordinación entre tarefas

Outro concepto importante para o control do fluxo son as `trigger rules`.

Por defecto, moitas tarefas executaranse só cando as súas tarefas previas remataron con éxito. Pero Airflow permite cambiar esa lóxica segundo o caso.

Isto é útil porque non todos os fluxos reais responden á regra simple de "seguir só se todo foi ben". Ás veces interesa:

- continuar aínda que unha rama concreta fose omitida
- lanzar unha tarefa de limpeza aínda que algo fallase
- unir ramas que non sempre se executan todas

As `trigger rules` permiten expresar estas situacións con máis precisión.

Desde o punto de vista didáctico, a idea importante non é memorizar todas as regras dispoñibles, senón entender que o avance dunha tarefa pode depender non só da orde, senón tamén do estado co que remataron outras tarefas.

Isto será especialmente útil cando aparezan DAGs con ramas, validacións ou tarefas finais de peche.

Algunhas das `trigger rules` máis habituais son estas:

- `all_success`: a tarefa só se executa se todas as tarefas previas remataron con éxito. É o comportamento por defecto.
- `all_failed`: a tarefa só se executa se todas as tarefas previas fallaron.
- `one_success`: abonda con que unha das tarefas previas remate con éxito.
- `one_failed`: abonda con que unha das tarefas previas falle.
- `none_failed`: a tarefa execútase se ningunha das tarefas previas fallou, aínda que algunha puidese quedar omitida.
- `none_failed_min_one_success`: a tarefa execútase se ningunha previa fallou e polo menos unha rematou con éxito.
- `all_done`: a tarefa execútase cando todas as tarefas previas remataron dalgunha maneira, con éxito, fallo ou omisión.

Nun primeiro contacto con Airflow non fai falta memorizar toda a lista, pero si convén recoñecer algúns patróns:

- `all_success` é a regra máis natural para fluxos lineais
- `none_failed_min_one_success` é moi útil para unir ramas despois dun `branching`
- `all_done` adoita ser útil en tarefas finais de limpeza ou peche

### Exemplo básico

O caso máis típico aparece precisamente despois dun `branching`. Se unimos dúas ramas nunha tarefa final, por defecto esa tarefa final pode quedar omitida porque unha das ramas foi marcada como `skipped`.

```python
from datetime import datetime

from airflow import DAG
from airflow.operators.empty import EmptyOperator
from airflow.operators.python import BranchPythonOperator
from airflow.utils.trigger_rule import TriggerRule


def escoller_rama(**context):
    executar_carga = context["params"]["executar_carga"]
    if executar_carga:
        return "tarefa_cargar"
    return "tarefa_omitir"


with DAG(
    dag_id="trigger_rules_demo",
    start_date=datetime(2026, 1, 1),
    schedule=None,
    catchup=False,
    params={"executar_carga": True},
    tags=["demo", "trigger-rules"],
) as dag:
    tarefa_decision = BranchPythonOperator(
        task_id="tarefa_decision",
        python_callable=escoller_rama,
    )

    tarefa_cargar = EmptyOperator(task_id="tarefa_cargar")
    tarefa_omitir = EmptyOperator(task_id="tarefa_omitir")
    tarefa_fin = EmptyOperator(
        task_id="tarefa_fin",
        trigger_rule=TriggerRule.NONE_FAILED_MIN_ONE_SUCCESS,
    )

    tarefa_decision >> [tarefa_cargar, tarefa_omitir]
    tarefa_cargar >> tarefa_fin
    tarefa_omitir >> tarefa_fin
```

Neste exemplo:

- unha das dúas ramas quedará omitida en cada execución
- coa regra por defecto, `tarefa_fin` tamén podería quedar omitida
- ao usar `none_failed_min_one_success`, `tarefa_fin` execútase se polo menos unha rama rematou ben e ningunha fallou

Este é un caso moi típico e moi útil para entender por que as `trigger rules` forman parte do control real do fluxo, especialmente cando o DAG deixa de ser unha cadea lineal simple.

## Como comprobar todo isto na práctica

Unha vez executado un DAG, o máis habitual é revisar a interface e os `Logs` para entender que ocorreu.

Segundo o mecanismo empregado, pode interesar mirar:

- os `Logs` da tarefa consumidora se se usou `XCom`
- os parámetros visibles na execución
- o resultado renderizado dun campo templado
- a rama finalmente seguida nun fluxo con `branching`
- o estado das tarefas cando hai `trigger rules` non triviais

Isto reforza unha idea moi importante en Airflow: non abonda con escribir o DAG; tamén hai que observar como se comporta realmente nunha execución concreta.

## Boas prácticas iniciais

Ao empezar a combinar comunicación, contexto e control do fluxo, convén manter algunhas regras simples:

- usar `XCom` só para valores pequenos
- non empregar Airflow como sistema principal de transporte de datos
- usar `params` cando o que se quere é configurar unha execución
- usar `Variables` para configuración compartida e relativamente estable
- usar `Connections` para separar do código o acceso a sistemas externos
- empregar `templates` cando realmente achegan claridade e reutilización
- manter o fluxo o máis lexible posible, especialmente cando hai ramas

Nun entorno docente, tamén convén introducir estes mecanismos de maneira progresiva. Primeiro interesa que o DAG se entenda ben; despois xa se pode engadir máis sofisticación.

## Resumo

Neste punto xa se pode ver que a comunicación entre tarefas en Airflow é máis ampla do que parece á primeira vista.

Non se limita a `XCom`. Tamén inclúe:

- dependencias que expresan orde
- parámetros que configuran a execución
- variables e conexións que separan configuración e acceso a sistemas
- templates que permiten usar valores dinámicos
- mecanismos de control do fluxo como `branching` e `trigger rules`

Nun proxecto como este, ademais, a idea clave é que Airflow coordina o fluxo, mentres que os datos principais adoitan vivir en sistemas externos como Delta ou Iceberg. Por iso, moitas veces a comunicación real entre tarefas consiste en combinar dependencias, pequenas referencias e decisións de control, máis que en transportar datos dentro de Airflow.

## Para seguir avanzando

Unha vez entendido este marco xeral, o seguinte paso natural é estudar operadores máis específicos cos que implementar estas ideas na práctica.

Nese punto xa non falaremos só de como se relacionan as tarefas, senón tamén de que ferramentas concretas ofrece Airflow para executar Python, shell, Spark e outros tipos de traballo dentro dun DAG.
