# Operadores específicos en Airflow

Nos capítulos anteriores vimos que un DAG define tarefas, dependencias e formas de comunicación entre pasos. O seguinte paso natural é entender mellor con que mecanismos concretos se implementan esas tarefas.

En Airflow, ese papel cúmpreno os `operators`. Un operador representa un tipo de execución: executar código Python, lanzar un comando do sistema, chamar unha API, poñer en marcha un job de Spark ou interactuar cun sistema de mensaxería, entre outras posibilidades.

Isto é importante porque, ao deseñar un DAG, non abonda con saber que tarefas queremos encadear. Tamén hai que decidir con que ferramenta concreta se vai executar cada paso.

Xunto coa idea de `operator`, neste capítulo tamén aparece varias veces o termo `provider`.

En Airflow, un `provider` é un paquete adicional que incorpora integracións cun sistema ou cunha tecnoloxía concreta. Eses paquetes poden incluír:

- operadores
- hooks
- sensores
- conexións e utilidades asociadas

Por exemplo:

- o soporte para Spark pode requirir o provider correspondente de Spark
- o soporte para Kafka pode requirir o provider correspondente de Kafka
- o soporte para chamadas HTTP pode requirir o provider correspondente de HTTP

Isto significa que non todos os operadores están dispoñibles do mesmo xeito nunha instalación mínima de Airflow. Algúns forman parte do uso máis básico da ferramenta, mentres que outros dependen de que o provider adecuado estea instalado.

Neste capítulo imos revisar varios operadores especialmente relevantes para este proxecto e para unha introdución práctica a Airflow:

- `PythonOperator`
- `BashOperator`
- `EmptyOperator`
- `SparkSubmitOperator`
- operadores de Kafka
  - `ProduceToTopicOperator`
  - `ConsumeFromTopicOperator`
- `HttpOperator`

Ao final engadiremos tamén unha nota breve sobre sensores, porque axudan a completar o panorama práctico, aínda que o seu papel é algo distinto.

## Que é un operador

Un operador é a peza que define como se vai executar unha tarefa.

Noutras palabras:

- a `task` é o paso concreto do DAG
- o `operator` é a peza de Airflow que define como se executa ese paso

Por exemplo:

- se queremos executar unha función Python, usaremos normalmente `PythonOperator`
- se queremos lanzar un comando do sistema, usaremos `BashOperator`
- se queremos enviar un job a Spark, usaremos `SparkSubmitOperator`

Desde un punto de vista didáctico, isto axuda a ver que Airflow non impón unha única forma de traballar. O mesmo DAG pode combinar tipos de tarefas moi distintos segundo o que precise cada paso.

## `PythonOperator`

`PythonOperator` é un dos operadores máis sinxelos e máis útiles para empezar. Permite executar unha función Python como parte dun DAG.

É especialmente adecuado para:

- exemplos introdutorios
- pequenas validacións
- transformacións simples
- lectura de parámetros ou contexto
- tarefas auxiliares de preparación

No contexto deste proxecto, adoita ter sentido en tarefas lixeiras, pero non como substituto de procesos pesados que deberían correr en Spark.

### Provider necesario

Non require un provider adicional específico para o enfoque deste capítulo. Forma parte dos operadores habituais dispoñibles en Airflow.

### Sintaxe básica

```python
from airflow.operators.python import PythonOperator


def miña_funcion():
    ...


tarefa = PythonOperator(
    task_id="miña_tarefa",
    python_callable=miña_funcion,
)
```

Os elementos principais son:

- `task_id`: nome interno da tarefa
- `python_callable`: función Python que se vai executar

### Exemplo sinxelo

```python
from datetime import datetime

from airflow import DAG
from airflow.operators.python import PythonOperator


def saudar():
    print("Ola desde PythonOperator")


with DAG(
    dag_id="python_operator_demo",
    start_date=datetime(2026, 1, 1),
    schedule=None,
    catchup=False,
    tags=["demo", "python-operator"],
) as dag:
    tarefa_python = PythonOperator(
        task_id="tarefa_python",
        python_callable=saudar,
    )
```

Neste exemplo, a tarefa simplemente executa unha función que imprime unha mensaxe. O resultado poderá verse nos `Logs` da task.

## `BashOperator`

`BashOperator` permite executar un comando de shell. É moi útil cando o paso do fluxo non consiste en código Python propio, senón nunha orde do sistema.

Isto resulta práctico para:

- lanzar scripts existentes
- consultar información do sistema
- encadear pequenas operacións de shell
- chamar utilidades dispoñibles no contorno do contedor

Hai que empregalo con criterio: funciona moi ben para tarefas simples e transparentes, pero se a lóxica comeza a facerse longa ou complexa, adoita ser mellor encapsulala doutro xeito.

### Provider necesario

Non require un provider adicional específico para o enfoque deste capítulo. Forma parte dos operadores habituais dispoñibles en Airflow.

### Sintaxe básica

```python
from airflow.operators.bash import BashOperator


tarefa = BashOperator(
    task_id="miña_tarefa",
    bash_command="echo 'Ola'",
)
```

O parámetro principal é:

- `bash_command`: comando que se executará no shell

### Exemplo sinxelo

```python
from datetime import datetime

from airflow import DAG
from airflow.operators.bash import BashOperator


with DAG(
    dag_id="bash_operator_demo",
    start_date=datetime(2026, 1, 1),
    schedule=None,
    catchup=False,
    tags=["demo", "bash-operator"],
) as dag:
    tarefa_echo = BashOperator(
        task_id="tarefa_echo",
        bash_command="echo 'Executando BashOperator en Airflow'",
    )
```

Neste caso, Airflow lanza un comando simple e a súa saída queda rexistrada nos logs da tarefa.

## `EmptyOperator`

`EmptyOperator` non executa traballo real. A súa utilidade está en estruturar o DAG e facer máis lexible o fluxo.

Pode empregarse para:

- marcar un punto de inicio ou de fin
- facer máis clara unha ramificación
- separar visualmente partes dun DAG
- introducir nodos de control sen lóxica de negocio

É un operador pequeno, pero moi útil cando os DAGs deixan de ser lineais e pasan a ter máis dunha rama.

### Provider necesario

Non require un provider adicional específico para o enfoque deste capítulo. Forma parte dos operadores habituais dispoñibles en Airflow.

### Sintaxe básica

```python
from airflow.operators.empty import EmptyOperator


tarefa = EmptyOperator(
    task_id="marcador",
)
```

### Exemplo sinxelo

```python
from datetime import datetime

from airflow import DAG
from airflow.operators.empty import EmptyOperator


with DAG(
    dag_id="empty_operator_demo",
    start_date=datetime(2026, 1, 1),
    schedule=None,
    catchup=False,
    tags=["demo", "empty-operator"],
) as dag:
    inicio = EmptyOperator(task_id="inicio")
    fin = EmptyOperator(task_id="fin")

    inicio >> fin
```

Aquí o operador non fai procesado, pero si axuda a representar a estrutura do fluxo.

## `SparkSubmitOperator`

`SparkSubmitOperator` é, con diferenza, un dos operadores máis relevantes neste proxecto. Permite lanzar aplicacións de Spark desde Airflow.

Isto encaixa directamente co papel de Airflow dentro do stack:

- Airflow coordina
- Spark executa o procesado pesado

Por iso, nun contorno como este, `SparkSubmitOperator` adoita ser máis representativo do uso real de Airflow que operadores máis simples pensados só para demostración.

### Provider necesario

Require o provider:

- `apache-airflow-providers-apache-spark`

No stack deste proxecto, este provider xa foi incorporado á imaxe de Airflow para poder traballar con Spark.

### Sintaxe básica

```python
from airflow.providers.apache.spark.operators.spark_submit import SparkSubmitOperator


tarefa = SparkSubmitOperator(
    task_id="spark_job",
    application="/opt/airflow/dags/jobs/exemplo.py",
    conn_id="spark_client_demo",
    spark_binary="/home/airflow/.local/bin/spark-submit",
    deploy_mode="client",
)
```

Os parámetros máis habituais son:

- `application`: ruta do script ou aplicación Spark
- `conn_id`: conexión de Airflow empregada para Spark
- `spark_binary`: binario de `spark-submit` que se vai empregar
- `deploy_mode`: modo de despregamento, por exemplo `client`
- `application_args`: argumentos que se lle pasan á aplicación
- `conf`: configuración adicional de Spark cando faga falta

### Configuración da conexión

Para os exemplos deste capítulo, resultou máis seguro empregar unha conexión distinta de `spark_default`, xa que a conexión por defecto do stack estaba condicionada por variables de contorno usadas nas probas previas.

A conexión validada para os exemplos foi:

- `Conn Id`: `spark_client_demo`
- `Conn Type`: `Spark`
- `Host`: `yarn`
- `Deploy mode`: `client`
- `Spark Binary`: `spark-submit`

Esta configuración permite que o operador use unha conexión limpa e máis controlada dentro da interface de Airflow, mentres que o parámetro `spark_binary` do propio operador pode seguir apuntando ao binario validado no contedor.

### Exemplo sinxelo

```python
from datetime import datetime

from airflow import DAG
from airflow.providers.apache.spark.operators.spark_submit import SparkSubmitOperator


with DAG(
    dag_id="spark_submit_demo",
    start_date=datetime(2026, 1, 1),
    schedule=None,
    catchup=False,
    tags=["demo", "spark-submit"],
) as dag:
    tarefa_spark = SparkSubmitOperator(
        task_id="tarefa_spark",
        application="/opt/airflow/dags/spark_apps/demo_spark.py",
        conn_id="spark_client_demo",
        spark_binary="/home/airflow/.local/bin/spark-submit",
        deploy_mode="client",
    )
```

Neste exemplo, Airflow non fai o procesado directamente. O que fai é lanzar unha aplicación Spark utilizando a conexión configurada.

Desde o punto de vista do proxecto, este é o operador que mellor representa a integración entre Airflow e o resto do stack de datos.

Para completar o exemplo, a aplicación Spark podería ser algo tan simple como isto:

```python
from pyspark.sql import SparkSession


def main() -> None:
    spark = SparkSession.builder.appName("demo-spark-airflow").getOrCreate()

    rows = [(1, "ana"), (2, "bruno"), (3, "carla")]
    df = spark.createDataFrame(rows, ["id", "nome"])

    print(f"SPARK_COUNT={df.count()}")
    df.orderBy("id").show(truncate=False)

    spark.stop()


if __name__ == "__main__":
    main()
```

Se este script estivese gardado en:

- `/opt/airflow/dags/spark_apps/demo_spark.py`

entón o DAG anterior podería lanzalo como unha aplicación Spark pequena de demostración.

## Operadores de Kafka

Airflow dispón tamén de operadores específicos para Kafka. Isto mostra que un DAG pode interactuar non só con procesos batch ou scripts, senón tamén con sistemas de mensaxería.

Nun curso ou nun proxecto introdutorio non sempre será o centro da práctica, pero é útil coñecer que existe esta posibilidade.

Os dous casos máis intuitivos para empezar son:

- producir mensaxes nun tópico
- consumir mensaxes desde un tópico

### Provider necesario

Requiren o provider:

- `apache-airflow-providers-apache-kafka`

Ademais do provider, estes operadores adoitan necesitar unha conexión de Airflow para Kafka. O identificador máis habitual é:

- `kafka_default`

Na interface de Airflow, o tipo de conexión aparece como:

- `Apache Kafka`

E, como mínimo, esa conexión debería incluír no campo de configuración ou `extra` algo equivalente a:

```json
{
  "bootstrap.servers": "kafka:9092"
}
```

No contorno deste proxecto, o valor validado foi `kafka:9092`, correspondente ao servizo Kafka dispoñible na rede Docker.

No caso do consumo de mensaxes, resultou útil ampliar os `extra` da conexión con máis configuración específica, por exemplo:

```json
{
  "bootstrap.servers": "kafka:9092",
  "group.id": "airflow-demo-consumer",
  "enable.auto.commit": false
}
```

Isto mostra unha idea importante: os operadores de Kafka non dependen só do provider, senón tamén dunha conexión ben configurada segundo o tipo de uso que se vaia facer.

En particular, ao consumir mensaxes convén ter en conta a relación entre:

- `group.id`
- offsets xa gardados en Kafka
- e o parámetro `auto.offset.reset`

Se un `group.id` xa ten offsets gardados, o consumer retomará desde esa posición e tenderá a ler só as mensaxes novas a partir dese punto. En cambio, se o grupo é novo ou non ten offsets válidos, entón `auto.offset.reset` pasa a ser determinante:

- `earliest`: intenta ler desde o inicio dispoñible
- `latest`: sitúase ao final e le só o novo que chegue despois

Por iso, para probas e exemplos, non abonda con crear a conexión: tamén convén entender se o consumer está a traballar cun grupo novo ou cun grupo que xa tiña historial previo.

### `ProduceToTopicOperator`

Este operador envía mensaxes a un tópico de Kafka.

#### Sintaxe básica

```python
from airflow.providers.apache.kafka.operators.produce import ProduceToTopicOperator


tarefa = ProduceToTopicOperator(
    task_id="producir",
    topic="demo_topic",
    producer_function="modulo.funcion_produtora",
)
```

#### Exemplo sinxelo

```python
from datetime import datetime

from airflow import DAG
from airflow.providers.apache.kafka.operators.produce import ProduceToTopicOperator


def producir_mensaxes():
    return [("clave-1", "mensaxe de proba")]


with DAG(
    dag_id="kafka_produce_demo",
    start_date=datetime(2026, 1, 1),
    schedule=None,
    catchup=False,
    tags=["demo", "kafka"],
) as dag:
    tarefa_producir = ProduceToTopicOperator(
        task_id="tarefa_producir",
        topic="demo_topic",
        producer_function=producir_mensaxes,
        kafka_config_id="kafka_default",
    )
```

Neste caso, a tarefa produce unha mensaxe de exemplo e envíaa ao tópico configurado.

### `ConsumeFromTopicOperator`

Este operador consume mensaxes dun tópico.

#### Sintaxe básica

```python
from airflow.providers.apache.kafka.operators.consume import ConsumeFromTopicOperator


tarefa = ConsumeFromTopicOperator(
    task_id="consumir",
    topics=["demo_topic"],
    apply_function="modulo.funcion_consumidora",
)
```

#### Exemplo sinxelo

```python
from datetime import datetime

from airflow import DAG
from airflow.providers.apache.kafka.operators.consume import ConsumeFromTopicOperator


def procesar_mensaxe(mensaxe):
    print(mensaxe)


with DAG(
    dag_id="kafka_consume_demo",
    start_date=datetime(2026, 1, 1),
    schedule=None,
    catchup=False,
    tags=["demo", "kafka"],
) as dag:
    tarefa_consumir = ConsumeFromTopicOperator(
        task_id="tarefa_consumir",
        topics=["demo_topic"],
        apply_function=procesar_mensaxe,
        kafka_config_id="kafka_default",
    )
```

Neste exemplo, a tarefa le mensaxes dun tópico e aplícalles unha función simple de procesado.

Desde o punto de vista do proxecto, estes operadores poden servir para mostrar que Airflow tamén pode integrarse con Kafka, aínda que o corazón do laboratorio siga estando máis ligado a Spark e ao almacenamento de datos.

## `HttpOperator`

`HttpOperator` é un operador moi útil porque funciona como peza xenérica para interactuar con servizos que expoñen unha API HTTP.

Isto é especialmente interesante cando:

- non existe un provider específico para o sistema que queremos chamar
- queremos lanzar unha petición REST simple
- precisamos integrar Airflow cun servizo externo de maneira lixeira

No contexto deste proxecto, é unha boa ponte conceptual para casos como Superset ou NiFi, onde unha integración básica pode facerse vía API sen necesidade dun operador moi especializado.

### Provider necesario

Require o provider:

- `apache-airflow-providers-http`

### Sintaxe básica

```python
from airflow.providers.http.operators.http import HttpOperator


tarefa = HttpOperator(
    task_id="chamada_http",
    http_conn_id="api_demo",
    endpoint="/status",
    method="GET",
)
```

Os elementos máis habituais son:

- `http_conn_id`: conexión HTTP definida en Airflow
- `endpoint`: ruta da API
- `method`: método HTTP, como `GET` ou `POST`
- `data` ou `headers`: cando faga falta enviar información adicional

### Configuración da conexión

Para as probas con NiFi, a conexión validada foi unha conexión HTTP co identificador:

- `nifi_default`

Coa configuración básica:

- `Conn Type`: `HTTP`
- `Schema`: `https`
- `Host`: `nifi`
- `Port`: `8443`

E no campo `extra`:

```json
{
  "verify": false
}
```

Isto foi necesario porque NiFi está exposto por HTTPS dentro da rede Docker e, para estas probas locais, interesaba desactivar a verificación do certificado.

### Exemplo 1: pedir un token a NiFi

Un primeiro uso realista de `HttpOperator` consiste en chamar ao endpoint de autenticación de NiFi para obter un token.

```python
from datetime import datetime

from airflow import DAG
from airflow.providers.http.operators.http import HttpOperator


with DAG(
    dag_id="nifi_token_demo",
    start_date=datetime(2026, 1, 1),
    schedule=None,
    catchup=False,
    tags=["demo", "http", "nifi"],
) as dag:
    pedir_token = HttpOperator(
        task_id="pedir_token",
        http_conn_id="nifi_default",
        endpoint="/nifi-api/access/token",
        method="POST",
        headers={"Content-Type": "application/x-www-form-urlencoded"},
        data="username=admin&password=Admin.BDA123!",
        log_response=True,
    )
```

Neste exemplo:

- Airflow chama á API de NiFi
- envía usuario e contrasinal como formulario
- a resposta contén un token de acceso

Este caso é útil porque mostra que `HttpOperator` non serve só para consultar estados, senón tamén para realizar chamadas de autenticación.

### Exemplo 2: usar o token nun `PythonOperator` para chamar á API

Unha vez obtido o token, pódese empregar noutra tarefa, neste caso implementada cun `PythonOperator`, para consultar un endpoint real de NiFi.

```python
from datetime import datetime

import requests
from airflow import DAG
from airflow.operators.python import PythonOperator


def pedir_token():
    response = requests.post(
        "https://nifi:8443/nifi-api/access/token",
        headers={"Content-Type": "application/x-www-form-urlencoded"},
        data="username=admin&password=Admin.BDA123!",
        verify=False,
    )
    response.raise_for_status()
    return response.text


def consultar_flow_status(ti=None):
    token = ti.xcom_pull(task_ids="pedir_token")
    response = requests.get(
        "https://nifi:8443/nifi-api/flow/status",
        headers={"Authorization": f"Bearer {token}"},
        verify=False,
    )
    response.raise_for_status()
    print(response.text)


with DAG(
    dag_id="nifi_api_demo",
    start_date=datetime(2026, 1, 1),
    schedule=None,
    catchup=False,
    tags=["demo", "nifi", "api"],
) as dag:
    tarefa_token = PythonOperator(
        task_id="pedir_token",
        python_callable=pedir_token,
    )

    tarefa_status = PythonOperator(
        task_id="consultar_flow_status",
        python_callable=consultar_flow_status,
    )

    tarefa_token >> tarefa_status
```

Neste segundo exemplo:

- a primeira tarefa obtén o token e devólveo
- Airflow gárdao temporalmente en `XCom`
- a segunda tarefa recupera ese token e úsao no header `Authorization`

Isto permite ver un patrón moi habitual na integración con APIs:

- primeiro autenticar
- despois chamar ao endpoint de interese

### Exemplo sinxelo de chamada HTTP directa


Neste caso, a tarefa realiza unha chamada HTTP simple empregando unha conexión xa configurada en Airflow.

## Sensores como complemento

Os sensores non son exactamente o mesmo que os operadores anteriores, porque a súa función principal non é executar unha acción, senón esperar por unha condición.

Con todo, é útil mencionalos aquí porque completan o modelo mental dos DAGs reais.

Un sensor pode empregarse para:

- esperar a que exista un ficheiro
- esperar a que responda unha API
- esperar a que apareza un recurso externo

Exemplos habituais son:

- `FileSensor`
- `HttpSensor`
- sensores específicos de HDFS, S3 ou outros sistemas segundo o provider instalado

Nun proxecto como este, teñen sentido cando un paso do fluxo non debe comezar ata que un recurso externo estea dispoñible.

## Como escoller operador no proxecto

Ao deseñar un DAG, unha boa pregunta práctica é: que tipo de tarefa é esta realmente?

Segundo a resposta, o operador máis natural cambia.

Pode servir esta guía simple:

- se o paso é unha función Python pequena, `PythonOperator`
- se o paso é un comando do sistema, `BashOperator`
- se o paso é só estrutural, `EmptyOperator`
- se o paso lanza procesado pesado en Spark, `SparkSubmitOperator`
- se o paso fala cunha API REST, `HttpOperator`
- se o paso interactúa con Kafka, operadores específicos de Kafka

Isto axuda a manter os DAGs máis claros e máis acordes coa responsabilidade real de cada tarefa.

## Boas prácticas iniciais con operadores

Ao empezar a traballar con distintos operadores, convén manter algunhas regras simples:

- escoller o operador máis simple que resolva ben o problema
- non usar `PythonOperator` para substituír procesos que deberían correr en Spark
- non meter comandos de shell complexos se sería máis claro encapsulalos noutro sitio
- revisar sempre que provider necesita cada operador
- comprobar que a conexión correspondente existe en Airflow cando o operador a precise
- empezar con exemplos pequenos antes de pasar a integracións máis ambiciosas

Nun contorno docente, isto é especialmente importante porque permite separar mellor:

- a lóxica do fluxo
- o mecanismo técnico de execución
- a integración cos sistemas externos

## Resumo

Neste capítulo vimos que os operadores son a peza que permite traducir un DAG a tarefas executables concretas.

Entre os máis importantes para este proxecto están:

- `PythonOperator`
- `BashOperator`
- `EmptyOperator`
- `SparkSubmitOperator`
- operadores de Kafka
- `HttpOperator`

Tamén vimos que algúns operadores forman parte do uso máis básico de Airflow, mentres que outros requiren providers adicionais, como ocorre con Spark, Kafka ou HTTP.

Desde o punto de vista práctico, a idea clave é esta:

- Airflow describe e coordina o fluxo
- os operadores materializan como se executa cada paso dese fluxo

## Para seguir avanzando

Unha vez entendidos os operadores máis importantes, o seguinte paso natural é estudar a dimensión temporal de Airflow:

- cando se executa un DAG
- que significa realmente `schedule`
- como inflúen `start_date`, `catchup` e a execución manual fronte á programada

Nese punto, Airflow xa deixa de verse só como un conxunto de tarefas e pasa a mostrarse como un sistema completo de planificación e orquestración.
