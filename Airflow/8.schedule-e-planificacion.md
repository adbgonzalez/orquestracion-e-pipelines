# `Schedule`, tempo e planificación en Airflow

Nos capítulos anteriores vimos como escribir DAGs, comunicar tarefas e empregar operadores concretos. O seguinte paso natural é entender cando se executa un DAG e como interpreta Airflow o tempo.

Esta parte adoita xerar bastante confusión ao empezar, porque Airflow non se limita a lanzar procesos "agora mesmo". A ferramenta traballa cunha idea máis formal da planificación:

- un DAG pode ter ou non ter calendario
- cada execución representa un intervalo lóxico de tempo
- `start_date`, `schedule` e `catchup` condicionan cando aparecen execucións
- unha execución programada non significa exactamente "executar nese instante", senón "executar o traballo correspondente a ese período"

Nun proxecto de datos isto é especialmente importante. Se un pipeline procesa datos diarios, non interesa só saber cando se puxo en marcha, senón que día de datos está procesando realmente.

## Por que o tempo importa en Airflow

Airflow é unha ferramenta de orquestración, e unha parte esencial de orquestrar consiste en decidir:

- cando se debe lanzar un fluxo
- con que frecuencia
- que intervalo temporal representa cada execución
- que facer coas execucións antigas pendentes

Nun exemplo simple, pode dar igual lanzar un DAG manualmente cando se queira. Pero nun pipeline real isto cambia moito.

Por exemplo, pode haber DAGs que:

- se executen cada día para procesar os datos do día anterior
- se executen cada hora para consolidar eventos recentes
- se executen só manualmente cando alguén queira probar algo
- se executen ao rematar outro fluxo

Por iso, entender a planificación non é un detalle secundario. É parte do modelo mental básico de Airflow.

## `schedule=None`: execución manual

A forma máis sinxela de empezar é definir un DAG con:

```python
schedule=None
```

Isto significa que o DAG non ten execución automática por calendario.

Na práctica:

- o DAG aparece na interface
- pode lanzarse manualmente
- non xera execucións periódicas por si só

Este enfoque é moi recomendable nos primeiros exemplos porque reduce bastante a complexidade temporal e permite centrarse na estrutura do DAG.

### Exemplo

```python
from datetime import datetime

from airflow import DAG


with DAG(
    dag_id="dag_manual_demo",
    start_date=datetime(2026, 1, 1),
    schedule=None,
    catchup=False,
    is_paused_upon_creation=False,
    tags=["demo", "manual"],
) as dag:
    pass
```

Este DAG:

- existe en Airflow
- pode executarse con `Trigger DAG`
- non lanzará execucións automáticas cada certo tempo

## `start_date`

`start_date` indica desde que punto temporal Airflow pode comezar a considerar execucións para ese DAG.

Isto non significa simplemente "momento no que se crea o DAG". Significa máis ben:

- desde que instante o calendario do DAG empeza a ter sentido

Por exemplo:

```python
start_date=datetime(2026, 1, 1)
```

indica que Airflow pode considerar execucións desde o 1 de xaneiro de 2026.

Se o DAG ten `schedule=None`, isto ten menos impacto práctico inmediato. Pero se hai un calendario definido, `start_date` pasa a ser esencial porque determina desde cando poden xerarse execucións programadas.

## `schedule`

O parámetro `schedule` define a frecuencia coa que Airflow debe crear execucións programadas para un DAG.

Segundo o caso, pode tomar formas como:

- `None`, se non hai calendario automático
- unha expresión cron
- un obxecto temporal, segundo o estilo empregado
- algúns presets ou atallos admitidos por Airflow

Nunha introdución práctica, o máis doado é pensar en tres grandes situacións:

- DAG manual: `schedule=None`
- DAG programado cun preset: por exemplo, `@daily` ou `@hourly`
- DAG programado cunha expresión cron: por exemplo, `"0 2 * * *"`

Isto é útil porque permite graduar a complexidade:

- `None` é a opción máis simple para aprender
- os presets serven para calendarios moi habituais sen ter que escribir cron
- cron serve cando necesitamos máis control

## Presets típicos de Airflow

Airflow inclúe varios valores abreviados moi comúns para `schedule`. Son prácticos porque evitan ter que lembrar a expresión cron equivalente nos casos máis habituais.

Os máis útiles ao empezar adoitan ser estes:

- `None`: o DAG non se programa automaticamente e queda para lanzamento manual ou externo
- `@once`: execútase unha soa vez
- `@continuous`: lánzase de novo en canto remata a execución anterior
- `@hourly`: unha vez por hora
- `@daily`: unha vez ao día
- `@weekly`: unha vez por semana
- `@monthly`: unha vez ao mes
- `@quarterly`: unha vez por trimestre
- `@yearly`: unha vez ao ano

Visto de forma máis concreta, a relación habitual entre preset e cron é esta:

- `@hourly` equivale a `0 * * * *`
- `@daily` equivale a `0 0 * * *`
- `@weekly` equivale a `0 0 * * 0`
- `@monthly` equivale a `0 0 1 * *`
- `@quarterly` equivale a `0 0 1 */3 *`
- `@yearly` equivale a `0 0 1 1 *`

Aquí convén lembrar unha convención moi útil ao ler cron: no campo de día da semana, `0` adoita representar o domingo e `1` o luns.

Desde un punto de vista docente, convén facer dúas precisións:

- `@daily` non significa "cada 24 horas desde o momento en que eu lancei o DAG", senón "cun calendario diario"
- `@hourly`, `@daily` e similares describen intervalos lóxicos de planificación, non só unha repetición mecánica

### Cando usar presets

Os presets adoitan ser a mellor opción cando:

- o calendario é simple e moi recoñecible
- queremos que o DAG sexa fácil de ler
- aínda non necesitamos unha regra máis específica

Por exemplo, se o obxectivo é dicir "este DAG é diario", `@daily` adoita comunicar mellor a intención que escribir directamente `0 0 * * *`.

### Exemplo diario con preset

```python
from datetime import datetime

from airflow import DAG


with DAG(
    dag_id="dag_diario_demo",
    start_date=datetime(2026, 1, 1),
    schedule="@daily",
    catchup=False,
    is_paused_upon_creation=False,
    tags=["demo", "schedule"],
) as dag:
    pass
```

Aquí o DAG queda configurado para unha execución diaria.

### Exemplo horario con preset

```python
from datetime import datetime

from airflow import DAG


with DAG(
    dag_id="dag_horario_demo",
    start_date=datetime(2026, 1, 1),
    schedule="@hourly",
    catchup=False,
    is_paused_upon_creation=False,
    tags=["demo", "schedule"],
) as dag:
    pass
```

Neste caso, Airflow xerará execucións horarias.

## Expresións cron

Unha forma moi habitual de definir calendarios en Airflow é usar expresións cron.

En Airflow, o formato máis habitual é o cron de cinco campos:

```text
minuto hora día_do_mes mes día_da_semana
```

Cada campo ocupa unha posición concreta:

- primeiro campo: minuto
- segundo campo: hora
- terceiro campo: día do mes
- cuarto campo: mes
- quinto campo: día da semana

Por exemplo, nesta expresión:

```python
schedule="0 2 * * *"
```

o significado é:

- `0`: no minuto 0
- `2`: ás 02:00
- `*`: calquera día do mes
- `*`: calquera mes
- `*`: calquera día da semana

É dicir:

- executar cada día ás 02:00

## Símbolos máis habituais en cron

Ao traballar con cron, os símbolos máis importantes ao empezar adoitan ser estes:

- `*`: calquera valor
- `,`: lista de valores
- `-`: rango
- `/`: salto ou frecuencia

### `*`: calquera valor

É o comodín básico.

Exemplo:

```python
schedule="0 * * * *"
```

Significa:

- no minuto 0 de cada hora

### `,`: lista de valores

Permite indicar varios valores concretos no mesmo campo.

Exemplo:

```python
schedule="0 9,17 * * *"
```

Significa:

- executar ás 09:00 e ás 17:00 cada día

### `-`: rango

Permite indicar un intervalo de valores.

Exemplo:

```python
schedule="0 9-17 * * *"
```

Interpretación básica:

- executar cada hora entre as 09:00 e as 17:00

### `/`: frecuencia ou salto

Permite definir intervalos regulares dentro dun rango.

Exemplo:

```python
schedule="*/15 * * * *"
```

Significa:

- executar cada 15 minutos

Outro exemplo:

```python
schedule="0 */2 * * *"
```

Significa:

- executar cada dúas horas, en punto

## Algúns valores cron típicos

Hai certas expresións que aparecen unha e outra vez e paga a pena recoñecer rapidamente:

- `0 * * * *`: cada hora en punto
- `0 0 * * *`: cada día á medianoite
- `0 2 * * *`: cada día ás 02:00
- `0 0 * * 1`: cada luns á medianoite
- `0 0 1 * *`: o día 1 de cada mes á medianoite
- `*/15 * * * *`: cada 15 minutos
- `0 */6 * * *`: cada 6 horas

Isto permite pasar dunha lectura teórica a unha lectura práctica do calendario.

## Como ler unha expresión cron paso a paso

Unha boa forma de non perderse é lela sempre de esquerda a dereita:

1. primeiro o minuto
2. despois a hora
3. despois o día do mes
4. despois o mes
5. finalmente o día da semana

Por exemplo:

```python
schedule="30 6 * * 1-5"
```

Pódese ler así:

- minuto `30`
- hora `6`
- calquera día do mes
- calquera mes
- días da semana de `1` a `5`

Interpretación práctica:

- executar ás 06:30, de luns a venres

## Cando usar cron en vez dun preset

As expresións cron permiten definir calendarios con bastante flexibilidade:

- por minuto
- por hora
- por día
- por mes
- por día da semana

Teñen sentido cando:

- o preset dispoñible non expresa exactamente o que queremos
- necesitamos unha hora concreta distinta da medianoite
- queremos limitar a certos días da semana
- necesitamos frecuencias como "cada 15 minutos" ou "cada 6 horas"

Nunha primeira aproximación non fai falla dominar todas as variantes. O importante é entender que Airflow pode empregar esta sintaxe cando se precisa un calendario máis concreto ca un simple `@daily`.

### Exemplo

```python
from datetime import datetime

from airflow import DAG


with DAG(
    dag_id="dag_cron_demo",
    start_date=datetime(2026, 1, 1),
    schedule="0 2 * * *",
    catchup=False,
    is_paused_upon_creation=False,
    tags=["demo", "cron"],
) as dag:
    pass
```

## Nota sobre Airflow e cron

Segundo a documentación oficial de Airflow, os DAGs poden programarse cunha expresión cron, cun `timedelta` ou cun dos presets de cron. Ademais, Airflow interpreta as expresións cron usando `croniter`, que soporta algunhas extensións máis avanzadas.

Isto significa que, para un uso introdutorio, chega con dominar o formato de cinco campos. Máis adiante, se se precisan calendarios especiais, xa pode ter sentido mirar as extensións concretas soportadas pola libraría.

## O punto máis importante: a execución representa un intervalo

Esta é probablemente a idea temporal máis importante de Airflow.

Cando un DAG ten calendario, unha execución programada non debe entenderse só como "un proceso lanzado a tal hora". Debe entenderse como a execución correspondente a un intervalo lóxico de datos ou de tempo.

Isto é moi natural en pipelines:

- unha execución diaria pode representar o día anterior
- unha execución horaria pode representar a hora anterior
- unha execución mensual pode representar o mes anterior

Desde o punto de vista didáctico, convén quedar con esta idea:

- Airflow non pensa só en instantes de lanzamento
- pensa tamén en períodos de procesado

Por iso, cando se revisan variables de contexto ou se fan particións de datos por data, ten moita importancia saber que intervalo está representando esa execución concreta.

## `logical_date` e intervalo de datos

Nas versións modernas de Airflow, aparece a idea de `logical_date`, moi ligada á identidade temporal dunha execución.

Sen entrar en excesivo detalle técnico, pode pensarse nela como:

- a referencia temporal coa que Airflow identifica esa execución

Isto é útil porque permite empregar o tempo de forma coherente dentro do DAG:

- para construír rutas
- para filtrar datos
- para xerar nomes de partición
- para rexistrar que intervalo se está procesando

Nun DAG, esta información pode aparecer en templates ou no contexto de execución.

### Exemplo con template

```python
from datetime import datetime

from airflow import DAG
from airflow.operators.bash import BashOperator


with DAG(
    dag_id="logical_date_demo",
    start_date=datetime(2026, 1, 1),
    schedule="@daily",
    catchup=False,
    is_paused_upon_creation=False,
    tags=["demo", "tempo"],
) as dag:
    mostrar_data = BashOperator(
        task_id="mostrar_data",
        bash_command="echo 'logical_date={{ ds }}'",
    )
```

Neste exemplo:

- `{{ ds }}` insire unha data derivada do contexto da execución
- esa data non é só un texto arbitrario
- está relacionada coa identidade temporal do `dag run`

## `catchup`

`catchup` indica se Airflow debe crear execucións antigas pendentes desde `start_date` ata o presente cando detecta que un DAG programado non foi executado nese intervalo.

Isto é unha idea moi importante.

Se un DAG ten:

- un `start_date` antigo
- un `schedule` definido
- e `catchup=True`

Airflow pode intentar crear moitas execucións pendentes do pasado.

Se, en cambio, ten:

- `catchup=False`

o comportamento é moito máis simple para empezar, porque Airflow non tenta poñerse ao día co historial completo desde a data inicial.

Xusto despois de decidir o `catchup`, adoita ser boa idea definir tamén como queres que quede o DAG ao crearse por primeira vez:

- `is_paused_upon_creation=False`

Con este parámetro, Airflow deixará o DAG en estado `unpaused` cando o vexa por primeira vez, polo que non terás que activar manualmente o interruptor na interface para que empece a planificarse.

Isto só afecta á creación inicial do DAG. Se o DAG xa existía e estaba pausado, engadir esta opción no código non o despausa automaticamente; nese caso hai que facer o `unpause` desde a UI ou coa CLI.


### Exemplo

```python
from datetime import datetime

from airflow import DAG


with DAG(
    dag_id="dag_con_catchup",
    start_date=datetime(2026, 1, 1),
    schedule="@daily",
    catchup=True,
    is_paused_upon_creation=False,
    tags=["demo", "catchup"],
) as dag:
    pass
```

Se este DAG se activa bastante despois do 1 de xaneiro de 2026, Airflow pode xerar execucións diarias pendentes desde esa data.

Agora comparemos con:

```python
from datetime import datetime

from airflow import DAG


with DAG(
    dag_id="dag_sen_catchup",
    start_date=datetime(2026, 1, 1),
    schedule="@daily",
    catchup=False,
    is_paused_upon_creation=False,
    tags=["demo", "catchup"],
) as dag:
    pass
```

Neste caso, o DAG queda moito máis controlado para unha introdución ou para probas locais.

## Cando usar `catchup=False`

En contornos docentes, de demostración ou de primeiras probas, adoita ser a opción máis cómoda.

Razóns:

- evita crear execucións antigas en bloque
- reduce confusión na interface
- facilita probar un DAG sen encher a historia de runs
- fai máis doado centrarse na lóxica do fluxo e non na recuperación do pasado

No contexto deste proxecto, isto encaixa especialmente ben nos exemplos introdutorios e nos DAGs de demostración.

## Cando podería ter sentido `catchup=True`

Hai casos reais nos que si pode ser útil.

Por exemplo:

- un pipeline diario que debe procesar cada día e non quere deixar ocos
- un DAG que se crea hoxe pero debe reconstruír execucións dos últimos días
- un fluxo que traballa con particións temporais e quere recuperar atraso

Mesmo así, convén empregalo con criterio. Se o DAG lanza tarefas pesadas, activar `catchup=True` cun `start_date` moi antigo pode provocar unha acumulación grande de execucións.

## Relación entre `start_date`, `schedule` e `catchup`

Estes tres elementos enténdense mellor xuntos que por separado.

### Caso 1: manual e sinxelo

```python
start_date=datetime(2026, 1, 1)
schedule=None
catchup=False
```

Interpretación:

- o DAG existe desde esa data de referencia
- non hai execucións automáticas
- lanzarase só manualmente

### Caso 2: diario sen recuperar pasado

```python
start_date=datetime(2026, 1, 1)
schedule="@daily"
catchup=False
```

Interpretación:

- o DAG ten unha frecuencia diaria
- non se intenta reconstruír todo o historial antigo
- adoita ser unha configuración boa para comezar

### Caso 3: diario con recuperación

```python
start_date=datetime(2026, 1, 1)
schedule="@daily"
catchup=True
```

Interpretación:

- o DAG é diario
- Airflow pode crear execucións pendentes desde a data inicial
- require máis coidado porque pode multiplicar o número de runs

## Exemplo completo dun DAG diario

```python
from datetime import datetime

from airflow import DAG
from airflow.operators.python import PythonOperator


def informar_contexto(**context):
    print(f"Logical date: {context['logical_date']}")
    print(f"Data ds: {context['ds']}")


with DAG(
    dag_id="schedule_diario_demo",
    start_date=datetime(2026, 1, 1),
    schedule="@daily",
    catchup=False,
    is_paused_upon_creation=False,
    tags=["demo", "schedule", "tempo"],
) as dag:
    tarefa_contexto = PythonOperator(
        task_id="tarefa_contexto",
        python_callable=informar_contexto,
    )
```

Este exemplo serve para unir varias ideas:

- o DAG ten frecuencia diaria
- non recupera automaticamente todo o pasado
- a tarefa pode acceder a información temporal do contexto

## Boas prácticas iniciais

Ao empezar con planificación en Airflow, convén seguir algunhas pautas simples:

- usar `schedule=None` nos primeiros DAGs de aprendizaxe
- empregar `catchup=False` mentres non haxa unha necesidade clara de reconstruír historial
- escoller un `start_date` consciente e non completamente arbitrario
- lembrar que unha execución programada representa un intervalo temporal, non só un instante
- usar o contexto temporal do DAG cando se traballa con rutas, particións ou datos por data

Nun stack de datos real, esta parte acaba sendo central. Cando un pipeline depende de datas, particións e execucións periódicas, entender ben o modelo temporal de Airflow evita moitos erros conceptuais e operativos.

## Resumo

Neste capítulo vimos que a planificación en Airflow non consiste simplemente en poñer unha tarefa a correr cada certo tempo. Tamén implica entender que:

- `schedule` define a frecuencia do DAG
- `start_date` marca desde cando ten sentido temporalmente
- `catchup` decide se se recuperan execucións antigas pendentes
- cada `dag run` ten unha identidade temporal propia
- esa identidade é moi relevante cando o pipeline procesa datos organizados por datas ou intervalos

Con isto xa temos unha base moito máis sólida para pasar de DAGs de demostración a fluxos programados con sentido temporal real.
