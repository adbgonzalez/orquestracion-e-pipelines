# Airflow no proxecto

Ata agora vimos que é Airflow e cales son os seus conceptos básicos. O seguinte paso é entender como encaixa dentro do entorno que estamos usando nas prácticas.

Neste caso, Airflow non aparece como parte do clúster base principal, senón como unha capa adicional de orquestración integrada sobre un stack modular de datos. O clúster de referencia empregado nas prácticas é:

- `https://github.com/adbgonzalez/big-data-cluster`

Ese repositorio define o entorno base sobre o que traballamos. Airflow engádese enriba dese stack para coordinar procesos e pipelines.

## O clúster base de referencia

O repo `big-data-cluster` está pensado como un clúster modular en Docker Compose orientado a docencia e probas locais. O núcleo do stack inclúe:

- Hadoop (`HDFS` + `YARN`)
- Spark
- JupyterLab

Ademais, pode ampliarse con módulos opcionais como:

- Kafka
- MinIO
- Apache NiFi
- Zeppelin
- Superset

A idea importante é esta: o clúster base fornece os servizos de almacenamento, procesado e traballo cos datos, mentres que Airflow se incorpora como compoñente de orquestración.

## Como encaixa Airflow neste entorno

Nunha arquitectura como esta, Airflow non substitúe Spark, nin MinIO, nin Kafka, nin HDFS. O seu papel é outro:

- definir fluxos de traballo
- establecer dependencias entre pasos
- decidir cando lanzar cada tarefa
- rexistrar estados, execucións e logs
- ofrecer unha interface para supervisar o pipeline

Por tanto, a relación conceptual entre compoñentes pode resumirse así:

- Spark procesa datos
- HDFS ou MinIO almacenan datos
- Kafka move eventos ou mensaxes
- Jupyter serve para exploración e desenvolvemento
- Airflow coordina a orde e o momento de execución

Esta separación é moi importante desde o punto de vista docente, porque axuda a entender que un pipeline real adoita construírse combinando varias ferramentas especializadas.

## Integración modular

Segundo as notas de apoio deste proxecto, Airflow intégrase de forma modular a través dun ficheiro específico:

- `compose.airflow.yml`

Isto segue a mesma filosofía do repo `big-data-cluster`, que xa organiza o stack en varios ficheiros Compose independentes para ir activando módulos segundo sexa necesario.

Así, a idea non é reconstruír o clúster desde cero, senón engadir Airflow como unha peza máis dentro do ecosistema.

No caso real deste laboratorio, ese ficheiro vive no outro repositorio local do clúster:

- `C:\repo\big-data-lab\big-data-cluster\compose.airflow.yml`

O ficheiro `compose.airflow.yml` empregado é este:


## Servizos de Airflow que forman parte do stack

Nas notas do proxecto descríbese unha integración de Airflow baseada en varios servizos separados. Os principais son os seguintes.

### `airflow-db`

É o contedor de Postgres que garda a metadata de Airflow.

Aquí almacénanse, entre outras cousas:

- definicións serializadas dos DAGs
- execucións dos DAGs
- estado das tarefas
- XComs
- información interna de operación

Esta base de datos non garda os datos de negocio do pipeline, senón o estado operacional de Airflow.

### `airflow-init`

É un servizo de inicialización pensado para a primeira posta en marcha.

O seu papel é preparar a base de datos e executar a inicialización necesaria de Airflow, especialmente a migración da metadata:

- `airflow db migrate`

Separar esta fase do arranque normal fai o proceso máis claro e evita mesturar bootstrap e execución diaria.

No `compose` real este servizo usa a mesma imaxe que o resto de compoñentes de Airflow:

- `adbgonzalez/airflow:3.1.8`

### `airflow-api-server`

É o servizo que expón a interface web e a API de Airflow.

Segundo a configuración real, o porto interno do servizo é `8080`, pero publícase cara ao host como:

- `${AIRFLOW_HOST_PORT:-8091}:8080`

Isto significa que, por defecto, o acceso se realiza en:

- `http://localhost:8091`

Desde esa interface pódense ver:

- os DAGs dispoñibles
- os `dag runs`
- o estado das tarefas
- os logs
- accións de lanzamento manual

### `airflow-scheduler`

É o compoñente que decide cando debe comezar un `dag run` e cando pode avanzar cada tarefa.

Ten en conta:

- o `schedule` definido no DAG
- as dependencias entre tarefas
- o estado de execución previo

### `airflow-dag-processor`

En Airflow 3 este compoñente ten un papel especialmente importante.

É o encargado de ler os ficheiros Python dos DAGs, extraer a súa definición e deixalos rexistrados para que o resto do sistema os coñeza.

Na práctica, cando se crea ou modifica un DAG, este servizo é unha das primeiras pezas que hai que mirar se o fluxo non aparece na interface.

## A imaxe Docker de Airflow neste laboratorio

Un detalle importante da integración é que non se usa só a imaxe oficial de Airflow, senón unha imaxe propia:

- `adbgonzalez/airflow:3.1.8`

Segundo o `Dockerfile` real, esa imaxe constrúese a partir dunha imaxe de Spark propia e da imaxe oficial de Airflow. O `Dockerfile` é este:

```dockerfile
FROM adbgonzalez/spark:4.1.1-py312 AS spark_runtime

FROM apache/airflow:3.1.8

USER root
RUN apt-get update && \
    apt-get install -y --no-install-recommends \
        openjdk-17-jre-headless \
        ca-certificates && \
    apt-get clean && \
    rm -rf /var/lib/apt/lists/*

COPY --from=spark_runtime /opt/spark /opt/spark
COPY --from=spark_runtime /usr/local/hadoop /usr/local/hadoop

ENV JAVA_HOME=/usr/lib/jvm/java-17-openjdk-amd64
ENV SPARK_HOME=/opt/spark
ENV HADOOP_HOME=/usr/local/hadoop
ENV HADOOP_CONF_DIR=/opt/hadoop-conf
ENV HADOOP_INSTALL=${HADOOP_HOME}
ENV HADOOP_MAPRED_HOME=${HADOOP_HOME}
ENV HADOOP_COMMON_HOME=${HADOOP_HOME}
ENV HADOOP_HDFS_HOME=${HADOOP_HOME}
ENV HADOOP_YARN_HOME=${HADOOP_HOME}
ENV HADOOP_COMMON_LIB_NATIVE_DIR=${HADOOP_HOME}/lib/native
ENV HADOOP_OPTS=-Djava.library.path=${HADOOP_HOME}/lib/native
ENV LD_LIBRARY_PATH=${HADOOP_HOME}/lib/native
ENV PATH=${JAVA_HOME}/bin:${HADOOP_HOME}/bin:${HADOOP_HOME}/sbin:${SPARK_HOME}/bin:${SPARK_HOME}/sbin:${PATH}

USER airflow

ARG AIRFLOW_VERSION=3.1.8
COPY requirements.txt /tmp/requirements.txt
RUN PYTHON_VERSION=$(python -c "import sys; print(f'{sys.version_info.major}.{sys.version_info.minor}')") && \
    CONSTRAINT_URL="https://raw.githubusercontent.com/apache/airflow/constraints-${AIRFLOW_VERSION}/constraints-${PYTHON_VERSION}.txt" && \
    pip install --no-cache-dir --constraint "${CONSTRAINT_URL}" -r /tmp/requirements.txt
```

E o `requirements.txt` asociado a esa imaxe contén:

```text
apache-airflow-providers-apache-spark==5.5.1
```

Isto significa que a imaxe de Airflow xa incorpora:

- Java 17
- Spark
- Hadoop
- o provider de Airflow para Apache Spark

O resultado é un contorno de Airflow preparado para lanzar tarefas que falen co stack de datos do laboratorio, especialmente con Spark sobre `YARN`.

## Rede e comunicación co resto do clúster

As notas do proxecto indican que Airflow se integra na mesma rede Docker do resto do stack:

- `cluster`

Isto é importante porque permite que os contedores de Airflow poidan comunicarse cos demais servizos empregando a rede interna de Docker Compose.

Nun escenario práctico, isto abre a porta a tarefas como:

- lanzar procesos Spark
- ler ou escribir en MinIO
- interactuar con servizos do clúster
- coordinar fluxos que dependan de compoñentes externos

Este detalle técnico é o que fai que Airflow non sexa só unha ferramenta illada, senón unha peza realmente conectada co ecosistema de datos do laboratorio.

No `compose` real hai, ademais, variables de contorno que deixan clara esa integración:

- `HADOOP_CONF_DIR=/opt/hadoop-conf`
- `SPARK_CONF_DIR=/opt/spark/conf`
- `AIRFLOW_CONN_SPARK_DEFAULT=spark://yarn?deploy-mode=client&spark-binary=%2Fhome%2Fairflow%2F.local%2Fbin%2Fspark-submit`

Isto deixa preparada unha conexión de Airflow cara a Spark usando `YARN` en modo cliente.

## Executor empregado

Segundo a configuración descrita nas notas, neste stack úsase:

- `LocalExecutor`

Esta decisión encaixa ben co contexto docente e local do proxecto porque:

- simplifica o despregamento
- evita engadir workers distribuídos e compoñentes extra como Redis
- permite certa concorrencia sen volver o sistema demasiado complexo

Isto tamén está aliñado co tipo de uso do repo `big-data-cluster`, que está pensado para probas locais e aprendizaxe, non para un despregamento de produción a gran escala.

## Volumes e cartafoles montados

As notas do proxecto describen varios montaxes importantes entre host e contedor:

- `./airflow/dags:/opt/airflow/dags`
- `./airflow/logs:/opt/airflow/logs`
- `./airflow/config:/opt/airflow/config`
- `./airflow/plugins:/opt/airflow/plugins`
- `./conf:/opt/hadoop-conf:ro`
- `./spark-conf/spark-defaults-airflow.conf:/opt/spark/conf/spark-defaults.conf:ro`

Isto permite distinguir ben varias responsabilidades:

- en `airflow/dags` escribimos os DAGs
- en `airflow/logs` quedan rexistrados os logs das tarefas
- en `airflow/config` pode gardarse configuración adicional
- en `airflow/plugins` poden vivir extensións ou compoñentes propios
- en `conf` móntase a configuración de Hadoop en modo só lectura
- en `spark-conf/spark-defaults-airflow.conf` móntase unha configuración específica de Spark para Airflow

Desde o punto de vista didáctico, este detalle é moi útil porque fai visible a separación entre:

- definición do pipeline
- execución do pipeline
- observabilidade
- configuración do sistema

Tamén mostra que Airflow reutiliza configuración real do clúster para poder traballar integrado con el.

## Fluxo de traballo dentro do proxecto

O fluxo xeral de traballo pode explicarse así:

1. o usuario escribe un DAG nun ficheiro Python
2. ese ficheiro queda dispoñible no cartafol montado de DAGs
3. o `airflow-dag-processor` interpreta a definición
4. o `airflow-scheduler` decide cando lanzar o fluxo
5. o `executor` executa as tarefas correspondentes
6. o `airflow-api-server` permite consultar estado e logs

Este percorrido é moi útil para aprender, porque permite relacionar cada paso conceptual con un compoñente concreto do sistema.

No caso deste laboratorio, a historia completa pode resumirse así:

1. o clúster base fornece `HDFS`, `YARN`, Spark e outros servizos
2. Airflow arranca na mesma rede Docker
3. os DAGs quedan montados en `/opt/airflow/dags`
4. o `dag-processor` rexístraos
5. o `scheduler` planifica os `dag runs`
6. as tarefas poden lanzar procesos que aproveiten Spark sobre `YARN`
7. os logs e estados quedan visibles na UI de Airflow

## Operación local simplificada

Nas notas tamén se describen varios scripts auxiliares para traballar con Airflow:

- `scripts/init-airflow.ps1`
- `scripts/up-airflow.ps1`
- `scripts/logs-airflow.ps1`
- `scripts/down-airflow.ps1`

O fluxo esperado sería:

### Primeira posta en marcha

1. executar `scripts/init-airflow.ps1`
2. executar `scripts/up-airflow.ps1`

### Uso normal

1. executar `scripts/up-airflow.ps1`
2. entrar na UI en `http://localhost:8091`

### Diagnóstico

- usar `scripts/logs-airflow.ps1` para revisar os logs dos servizos

### Parada

- usar `scripts/down-airflow.ps1`

## Que mirar cando algo non funciona

Neste proxecto resulta útil distinguir tres niveis de diagnóstico.

### Logs da tarefa

Son os primeiros que hai que mirar cando un DAG aparece pero falla unha task. Aquí adoitan verse:

- saídas de `print(...)`
- saída de comandos
- erros concretos da execución

### Logs dos servizos de Airflow

Son os máis útiles cando o problema é estrutural:

- o DAG non aparece
- o DAG non se parsea ben
- o scheduler non o lanza
- hai problemas internos de execución

Os máis importantes neste caso adoitan ser:

- `airflow-dag-processor`
- `airflow-scheduler`
- `airflow-api-server`

### Cartafol de DAGs dentro do contedor

Se un DAG novo non aparece, unha comprobación práctica é verificar que o ficheiro está visible dentro de:

- `/opt/airflow/dags`

Isto axuda a distinguir entre:

- problema de mount
- problema de parseo
- simple atraso no procesado

## Unha idea clave para situarse

O máis importante deste capítulo non é memorizar nomes de servizos, senón entender a lóxica da integración:

- o repo `big-data-cluster` achega o entorno de datos base
- Airflow engádese como capa de orquestración
- ambos conviven na mesma infraestrutura Docker
- os DAGs serven para coordinar o uso dos distintos compoñentes do stack
- a imaxe de Airflow xa está preparada para falar con Spark e Hadoop

Visto así, Airflow non é unha peza illada, senón o punto desde o que podemos comezar a unir almacenamento, procesado e automatización nun fluxo coherente.
